Edit `src-tauri/src/saved_searches/mod.rs`: add `pub mod commands;` alongside the existing `pub mod store;`.

Create `src-tauri/src/saved_searches/commands.rs` implementing four `#[tauri::command] async fn`s. Every command MUST:
- Take `state: State<'_, AppState>` as the last parameter.
- Clone the `Arc<Mutex<SavedSearchStore>>` from `state` into a local before entering `spawn_blocking`.
- Do all filesystem work (`SavedSearchStore::save`) inside `tokio::task::spawn_blocking` — mirrors `commands::entities` convention.
- Return `Result<T, AppError>`.

The four commands:

(1) `pub async fn get_saved_searches(state) -> Result<Vec<SavedSearch>, AppError>`:
   - Acquire the store mutex, clone `store.all()` into a `Vec<SavedSearch>`, return it.
   - No filesystem write, no spawn_blocking needed (pure in-memory clone).

(2) `pub async fn save_search(name: String, query: String, filters: SavedSearchFilters, state) -> Result<SavedSearch, AppError>`:
   - Generate `id = format!("ss-{}", uuid::Uuid::new_v4())`.
   - Compute `created_at = chrono::Utc::now().to_rfc3339()` (use `chrono` if already in Cargo.toml; otherwise use `std::time::SystemTime` + `humantime` or an inline formatter — check `rg chrono src-tauri/Cargo.toml` and prefer the existing timestamp path used by `entity_store` or `space_label_cache`).
   - Compute `doc_count_cache` inline: call the same metadata+entity-filter count logic used by `get_saved_search_counts` (extract it into a private `count_matching_docs` helper — see command 4). This gives a correct initial count without a follow-up sidebar-mount round trip.
   - Build `SavedSearch { id, name, query, filters, created_at, doc_count_cache }`.
   - Clone the store `Arc`, `spawn_blocking` to lock, `store.insert(new_search.clone())`, `store.save(&app_data_dir)`, propagate errors as `AppError::Internal`.
   - Return the new `SavedSearch`.

(3) `pub async fn delete_saved_search(id: String, state) -> Result<(), AppError>`:
   - Same spawn_blocking pattern: lock, `store.remove(&id)`, `store.save(&app_data_dir)`.
   - If remove returns false (id not found), return `AppError::NotFound(format!("saved search not found: {}", id))`.

(4) `pub async fn get_saved_search_counts(ids: Vec<String>, state) -> Result<HashMap<String, u32>, AppError>`:
   - Per Open Q 1 resolution in 11-RESEARCH.md: **metadata-only count, no embedding call, no HNSW.** This keeps Sidebar mount performant on personal-corpus scale.
   - Inside spawn_blocking, lock the saved_search_store, entity_store, and engine (engine uses `tokio::sync::Mutex` — use `.blocking_lock()` since we're inside spawn_blocking, mirror `commands::entities::get_documents_for_entity`).
   - For each `id`:
     - Look up the SavedSearch by id; if missing, insert 0 into the result map.
     - Parse `filters.entities` (Vec<String> of `"{class}:{value}"`) into `Vec<EntityClassFilter>` via `split_once(':')`.
     - Build a synthetic `SearchFilters` with `entity_filters: Some(parsed)`, `doc_type: filters.doc_type.clone()`, `space_id: filters.space_id.clone()`, `date_from/date_to/tags` copied over.
     - Call the private `count_matching_docs(&search_filters, &entity_store, &engine)` helper (Task 1 defines it — extracted so both `save_search` and this command use the same logic).
     - Insert `(id, count)`.
   - Return the HashMap.
   - The `count_matching_docs` helper composes:
     - `apply_metadata_filters(&search_filters, engine)` → metadata_candidate_set
     - `apply_entity_class_filters(entity_filters, entity_store)` → entity_candidate_set
     - Combine via intersection per Plan 03's table.
     - Return `combined.map(|s| s.len() as u32).unwrap_or_else(|| /* count all docs in collection */)`.

Add unit tests where feasible:
- Test `count_matching_docs_empty_filters_returns_all` — seeded engine + entity_store, no filters, returns total doc count.
- Test `count_matching_docs_entity_filter_narrows` — seeded engine with 5 docs, entity filter matches 2 → returns 2.
- Test `save_search_generates_ss_prefix_uuid` — parse returned id, assert `starts_with("ss-")` and remainder is a valid UUID.

Use `#[tokio::test]` where async is required; use `tempfile::TempDir` for filesystem work.

Register commands via `pub use commands::*;` in `mod.rs` if convenient, OR call them via `saved_searches::commands::save_search` in `lib.rs` (Task 3).

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| frontend → save_search | User-supplied `name` and `query` strings arrive from IPC |
| frontend → delete_saved_search | User-supplied `id` string arrives from IPC |
| filesystem → SavedSearchStore | app_data_dir may be tampered by another process |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-11-10 | Input Validation | save_search accepts empty name | mitigate | Reject empty/whitespace-only names via `if name.trim().is_empty() { return Err(AppError::Internal("...")) }` at command entry |
| T-11-11 | Denial of Service | get_saved_search_counts called with 10,000 ids | accept | Personal corpus scale; single query for id list is bounded by number of saved searches (typical < 20) |
| T-11-12 | Deadlock | entity_store std::sync::Mutex held across await | mitigate | All entity_store locks are inside spawn_blocking; pitfall #1 in 11-RESEARCH.md |
| T-11-13 | Torn write | Concurrent save_search + delete_saved_search interleaved | mitigate | Single `Arc<tokio::sync::Mutex<SavedSearchStore>>` serializes all mutations |
</threat_model>
