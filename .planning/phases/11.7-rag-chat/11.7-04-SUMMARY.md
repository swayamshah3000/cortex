# Phase 11.7 Plan 04: Streaming AI Service Summary

**`ai_request_stream()` central dispatch returning `Stream<StreamChunk>`, with pure SSE/NDJSON parsers + async wrappers for Anthropic, OpenAI, Codex, Gemini, and Ollama (with non-streaming fallback), built on `async-stream` + `reqwest::Response::bytes_stream()`.**

## Accomplishments

- `StreamChunk` enum (`Token { token, cumulative_index }`, `Done { input_tokens, output_tokens, model }`, `Error { message }`) — the wire contract consumed by the future ChatEngine (Plan 11.7-05) to emit `chat-stream-token` / `chat-stream-complete` / `chat-stream-error` Tauri events.
- `ai_request_stream()` mirrors `ai_request()`'s exact credential-resolution, codex-intercept, and provider-normalization logic — same preflight OAuth refresh, same "No AI provider configured" / "No credentials stored" error messages, same model-override fallback chain.
- All four providers' SSE/NDJSON wire formats parsed by **pure, unit-tested functions** (`parse_anthropic_sse_frames`, `parse_openai_sse_frames`, `parse_codex_sse_frames`, `parse_ollama_ndjson_frames`, `parse_gemini_sse_frames`) — no network I/O required to test parsing logic.
- Ollama fallback: if the local server ignores `stream: true` and returns one monolithic JSON object, the same NDJSON parser still emits exactly one `Token` + one `Done` (no panics, no special-casing needed).
- Gemini emits `Done { input_tokens: None, output_tokens: None }` when the stream ends without a `usageMetadata` frame — no fabrication, per the plan's zero-fabrication requirement.
- 1 MiB partial-buffer cap in every async wrapper mitigates the DoS threat (T-11.7-08): a malformed/never-terminated frame from a compromised endpoint aborts the stream with an `Error` chunk instead of growing memory unbounded.
- Zero regression: all 15 pre-existing `ai::service` tests and all pre-existing `ai::anthropic` / `ai::openai` tests still pass.

## Decisions Made

- **async-stream over tokio-stream/mpsc:** The plan offered a choice; `async-stream::stream!` was chosen for its generator-style syntax that keeps the SSE/NDJSON incremental-parsing logic linear and readable, and it's a smaller dependency than pulling in the full `tokio-stream` helper surface.
- **No 401 retry for streaming:** Documented explicitly in the module doc comment and every async wrapper's function doc comment. Consistent with the plan's T-11.7-11 "accept" disposition — SSE byte-stream replay after partial flush is non-trivial and deferred to v1.3.
- **SSE frame formats (reference for downstream debugging):**
  - **Anthropic:** `event: <type>\ndata: {json}\n\n`. Types used: `message_start` (model + input_tokens), `content_block_delta` (delta.type=="text_delta" → delta.text), `message_delta` (usage.input_tokens/output_tokens), `message_stop` (terminal), `error` (error.message).
  - **OpenAI (Chat Completions):** `data: {json}\n\n`, terminated by `data: [DONE]\n\n`. Token: `choices[0].delta.content`. Usage only appears in the final chunk (requires `stream_options.include_usage: true` in the request body) as a top-level `usage` object with `choices: []`.
  - **Codex (Responses API) `[ASSUMED]`:** `data: {json}\n\n` (no `event:` field observed in the OpenAI-published pattern; parser also checks `json.type` as a fallback). Types: `response.output_text.delta` (delta string), `response.completed` (response.usage.input_tokens/output_tokens), `response.failed` (response.error.message). **Not yet verified against live wire bytes** — flagged in code doc comment for easy correction if the actual shape differs.
  - **Gemini:** `data: {json}\n\n` via `?alt=sse` query param (explicitly required — default is a JSON array without this param). Token: `candidates[0].content.parts[0].text`. Usage: top-level `usageMetadata.promptTokenCount` / `usageMetadata.candidatesTokenCount`, typically only on the final frame.
  - **Ollama:** NDJSON (newline-delimited JSON objects, NOT SSE — no `data:` prefix, no blank-line frame separator). Each line: `{"message":{"content":"..."},"done":false}` for deltas, final line `{"done":true,"prompt_eval_count":N,"eval_count":M}`. Non-streaming fallback: single line with both `message.content` and `done:true` set.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Fixed non-compiling integration test from plan spec**
- **Found during:** Task 5 (ai_request_stream central dispatch)
- **Issue:** The plan's suggested `dispatch_tests` integration test called `result.unwrap_err()` on a `Result<Pin<Box<dyn Stream<...> + Send>>, String>`. `Pin<Box<dyn Stream>>` does not implement `Debug`, so `unwrap_err()` (which requires the `Ok` variant to be `Debug` for its panic message) fails to compile with `E0277`.
- **Fix:** Replaced `assert!(result.is_err()); assert!(result.unwrap_err().contains(...))` with an explicit `match result { Err(e) => assert!(...), Ok(_) => panic!(...) }` that never needs `Debug` on the `Ok` type.
- **Files modified:** `src-tauri/src/ai/stream.rs`
- **Verification:** `cargo test --lib ai::stream::dispatch_tests` passes; `cargo check --lib` shows zero errors.
- **Committed in:** `3401690` (Task 5 commit)
