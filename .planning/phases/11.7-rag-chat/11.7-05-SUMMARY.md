## How retrieval maps HNSW ids -> doc text (v1)

Per the plan's `[ASSUMED]` note: document text for chunking comes exclusively from `metadata["excerpt"]` on each `documents_384` HNSW hit. A document without an `excerpt` field (or an empty one) is silently skipped — not an error, not a zero-score inclusion. This is the same excerpt-only assumption already used by `search::query::search_documents_impl` for search-result highlighting. A "full text" store is out of scope for v1 (noted in the PLAN.md as v1.3 follow-up).

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Citation doc_title fell back to raw doc_id instead of resolving metadata title**
- **Found during:** Task 2 initial implementation (before either engine.rs or commands.rs commit landed)
- **Issue:** The first draft of `retrieve_and_rerank`/`answer` used `doc_id.clone()` as a placeholder for `doc_title` in `Citation` and the numbered-docs prompt, which would have shown raw ids like `doc-a1b2c3` in the chat UI's citation chips instead of readable titles like `Deed.pdf`.
- **Fix:** `retrieve_and_rerank` now reads `metadata.get("title")` per HNSW hit (falling back to `"Unknown"`, the same convention `build_document_from_metadata` in `search/query.rs` already uses), threading the title through the internal `(doc_id, doc_title, Chunk, score)` tuple to both `Citation.doc_title` and the RAG prompt's numbered-docs list.
- **Files modified:** `src-tauri/src/chat/engine.rs`
- **Commit:** 3ad5384 (bundled with Task 3's commands.rs commit, since the fix was made before Task 2's placeholder ever landed in a prior commit — engine.rs's Task 2 commit (cbe4d7d) already reflects an earlier revision; the title-resolution fix itself is isolated in 3ad5384's diff of engine.rs)
