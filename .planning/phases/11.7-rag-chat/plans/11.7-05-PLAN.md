**2. ChatEngine facade:**

```rust
pub struct ChatEngine;

impl ChatEngine {
    /// Runs the full RAG pipeline for a single query.
    ///
    /// Steps:
    /// 1. Embed query via EmbeddingService.
    /// 2. HNSW top-8 docs from documents_384 collection, applying optional filters
    ///    (mirror search_documents_impl candidate-set logic for D-04).
    /// 3. For each top doc, load its text (from raw.metadata "excerpt" for v1 — see [ASSUMED] note),
    ///    chunk 500/50-overlap, embed each chunk, keep top-3 by cosine to query embedding.
    /// 4. Combine into 24 candidates, rerank to top-12.
    /// 5. If best-chunk score < 0.35 → return canned answer path (no LLM call).
    /// 6. Build RAG prompt + call ai_request_stream, forwarding StreamChunk items
    ///    to the Tauri event bus.
    /// 7. On completion, persist assistant ChatMessage (with citations) to store.
    pub async fn answer(
        app: tauri::AppHandle,
        auth: Arc<AuthState>,
        engine: Arc<Mutex<CortexEngine>>,
        embedding_service: Arc<EmbeddingService>,
        entity_store: Arc<std::sync::Mutex<EntityStore>>,
        chat_store: Arc<Mutex<ChatSessionStore>>,
        app_data_dir: PathBuf,
        session_id: String,
        user_message_id: String,
        assistant_message_id: String,
        query: String,
        filters: Option<SearchFilters>,
    ) -> Result<(), String>;
}
```

**Retrieval details:**

- HNSW k=8 (not 20 like search). If filters provided, use `apply_metadata_filters` + `apply_entity_class_filters` from `crate::search::filters` (same combine truth-table as search_documents_impl) to build candidate_set, then intersect with the HNSW results.
- For each of the 8 doc ids, extract text from `raw.metadata["excerpt"]` as a `&str`. **[ASSUMED]:** Excerpt is the only readily-available text in Cortex metadata for v1. If a doc has no excerpt, skip it (log and move on). A follow-up plan can add a "full text" store; that is v1.3.
- Chunk each doc's excerpt via `chunk_text(text, 500, 50)`.
- Embed each chunk via `embedding_service.embed_text(chunk.text)` — synchronous std::sync::Mutex; wrap the whole retrieval step in `tokio::task::spawn_blocking`.
- Score each chunk by `cosine_sim(query_vec, chunk_vec)`. Keep top-3 per doc.
- Collect all top-3s into a single Vec; call `rerank_chunks(candidates, 12)`.

**Below-floor branch (D-03):**

- If `answer_or_canned(best_chunk_score)` returns Some, do:
  - Emit a single `chat-stream-token` with the canned message.
  - Emit `chat-stream-complete` with `citations: []`, `input_tokens: None`, `output_tokens: None`.
  - Persist the assistant ChatMessage (content = canned, citations = None) to store.
  - Return Ok(()).

**Streaming branch:**

- Build citations `Vec<Citation>` from top-12 (index = 1..=12, chunk_start / chunk_end from Chunk, doc_id + doc_title from metadata).
- Build numbered_docs `Vec<(u32, &str, &str)>` for the prompt.
- Call `crate::ai::stream::ai_request_stream(&auth, request)` where `request.system_prompt = RAG_SYSTEM_PROMPT`, `request.messages = vec![ServiceMessage { role: "user", content: build_rag_prompt(...) }]`, `request.max_tokens = Some(2048)`, `model_override = None`.
- For each stream item:
  - `Token { token, cumulative_index }` → append to a `String content_buffer` AND emit `chat-stream-token` with the payload.
  - `Done { input_tokens, output_tokens, model }` → emit `chat-stream-complete` with `citations` from the numbered_docs list, then persist the assistant ChatMessage (content = content_buffer, citations = Some(citations)).
  - `Error { message }` → emit `chat-stream-error` and persist a truncated assistant message with the error string as content (so the user sees what failed on reload). Return Err(message).

**Session bookkeeping:**

- Before the pipeline starts, the CALLER (Task 3) is expected to have already called `create_session` if needed and appended the user's ChatMessage. `ChatEngine::answer` only appends the assistant's response.

**Concurrency:** The whole `answer` fn is async. Any std::sync::Mutex guard (entity_store) MUST be dropped before an `.await`. Use spawn_blocking for the retrieval segment.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Frontend ↔ start_chat IPC | Untrusted query string + optional session_id + optional filters. |
| ChatEngine ↔ LLM provider | Cortex's system prompt + user's numbered-doc prompt. LLM cannot see credentials. |
| ChatEngine ↔ ChatSessionStore | Concurrent writes serialized by tokio Mutex. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-11.7-12 | Denial of Service | User submits 100k-char query → 100k-char embedding call → huge prompt | mitigate | `embed_text` already truncates to 2000 chars. Additionally cap query length in start_chat: `if query.len() > 4000 return Err`. |
| T-11.7-13 | Tampering | User-supplied session_id references a session belonging to a different user | accept | Cortex is single-user desktop; there is no multi-tenant boundary. Documented posture. |
| T-11.7-14 | Information Disclosure | RAG prompt leaks full chunks of every top-8 doc to the active LLM provider | accept | This is the explicit user consent contract from Phase 7: user chose an AI provider and knows queries + document snippets flow to it. Documented in PROJECT.md privacy stance. |
| T-11.7-15 | Elevation of Privilege | Prompt injection in a document chunk causes the LLM to ignore RAG_SYSTEM_PROMPT and reveal something outside the corpus | mitigate | RAG_SYSTEM_PROMPT (D-06) explicitly forbids fabrication + citing sources not in the list. The LLM's zero-fabrication instruction reduces (does not eliminate) injection impact. v1 posture: user's query is not privileged, so a compromised chunk can at worst produce a wrong answer, not exfiltrate credentials or files. Future hardening (input sanitization, separator tokens) is v1.3. |
| T-11.7-16 | Repudiation | Chat history is not audited or signed | accept | Personal desktop tool; no audit requirement. |
</threat_model>
