# Phase 11.6 Plan 02: OntologyStore JSON Sidecar Summary

**New `graph/ontology_store.rs` module: JSON-sidecar-backed adaptive vocabulary store with forward/reverse indices, a min-support (2 distinct docs) promote-pending gate bounded by a 200-predicate hard cap, seed-predicate immutability enforcement, and an `apply_consolidation` hand-off returning typed `TripleRewriteInstruction`s for the caller to apply to `TripleStore`.**

## Accomplishments
- `OntologyStore::load`/`save` implement the exact `SavedSearchStore`/`TripleStore` JSON-sidecar contract: silent-fail-to-default on I/O/parse error, atomic `.json.tmp` + rename on save.
- `effective_predicates()` deterministically merges `SEED_PREDICATES` (source=Seed, sentinel count=`u32::MAX`) + `corpus_seed` + `manual_predicates` + `adaptive_predicates`, deduping by name with earlier layers always winning and insertion order preserved for LLM prompt determinism (D-05).
- `record_pending_predicate()` implements the D-06/D-08 promotion gate: first occurrence stays pending (count=1); second distinct-doc occurrence promotes to `adaptive_predicates` only while `effective_predicate_names().len() < VOCABULARY_HARD_CAP` (200), otherwise the entry remains parked in `pending_predicates` and `PromoteResult::CapExceeded` is returned (no data loss).
- `apply_bootstrap()` is idempotent (no-op if `bootstrap_completed_at` is already set) and skips any corpus-seed predicate whose name collides with a seed predicate (D-03).
- Seed-predicate immutability enforced at the API boundary: `rename_predicate()` and `merge_predicates()` reject any seed-set name with `AppError::Invalid`, verified by dedicated tests.
- `apply_consolidation()` mutates the ontology's own vocabulary (via the existing rename/merge/register_manual_predicate methods) and returns `Vec<TripleRewriteInstruction>` — the store never reaches into `TripleStore` directly, keeping the two JSON-sidecar stores decoupled per the plan's design.
- 20 unit tests covering load resilience (missing file, corrupt JSON), round-trip persistence, dedup/order, index rebuild-on-load, the full promotion state machine (pending -> promoted -> cap-exceeded), bootstrap idempotency + seed-dup skip, manual-predicate snake_case validation, seed immutability on rename/merge, reset-to-seed, and consolidation apply + suggestion removal.

## Decisions Made
- `AppError::Invalid(String)` added to `error.rs` — the plan explicitly calls for `Err(AppError::Invalid(...))` in `rename_predicate` and validation failures, but that variant did not exist in the 8-variant `AppError` enum. Added it in the same tagged-enum style (`#[error("Invalid input: {0}")]`) as the other variants; no architectural change, purely additive.
- `name_to_source` intentionally does not index `pending_predicates` — only effective-vocabulary members (seed/corpus/manual/adaptive) are indexed there, matching the plan's description of the forward index as "predicate name -> source" for the *effective* vocabulary. Pending lookups go through `schema.pending_predicates` directly.
- `merge_predicates` sums `count` across all merged `from` entries and adds the total to whichever list (adaptive, then manual, then pending, checked in that order) contains the surviving `into` predicate — implements the plan's "into.count += from_entry.count" instruction for the general N-into-1 merge case.
- `apply_consolidation`'s `Split` branch returns `TripleRewriteInstruction { to: String::new() }` exactly as specified — the plan is explicit that Phase 11.6 does not auto-rewrite splits, only records the vocabulary change and defers actual triple reassignment to the user.
