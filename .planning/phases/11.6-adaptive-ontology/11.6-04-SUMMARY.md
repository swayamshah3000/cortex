# Phase 11.6 Plan 04: Corpus-Seeded Ontology Bootstrap Summary

Ships the "cortex wakes up smart" moment: after the first 30 documents complete Pass 2 successfully, a single LLM call proposes 15-30 domain-specific predicates + 5-15 entity subclasses from sampled titles/topics/tags/entities, validated and merged into `OntologyStore.corpus_seed`.

## What Was Built

**`src-tauri/src/pipeline/ontology_bootstrap.rs`** (703 lines, new module):
- `BOOTSTRAP_PROMPT_VERSION = "v1"` and `BOOTSTRAP_PROMPT` const (~50-line system prompt + few-shot example). Explicitly enumerates all 21 Phase 11.5 seed predicates in a "do NOT propose" block so the LLM never emits a collision.
- `BootstrapSampleDoc { title, topic, tags, top_entities }` — plain input struct, no serde needed (constructed in-process by `backfill.rs`).
- `RawBootstrapPredicate` / `RawBootstrapSubclass` / `RawBootstrapOutput` — internal camelCase deserialize targets for the raw LLM JSON response.
- `parse_bootstrap_json(raw) -> Result<RawBootstrapOutput, AppError>` — two-attempt parse (direct, then fence-stripped via `pass2_llm_refiner::strip_json_fences` reuse).
- `validate_bootstrap(...)` — drops seed-name collisions, invalid/non-snake_case/oversized predicate names, in-batch duplicates, caps at remaining `VOCABULARY_HARD_CAP` headroom; drops invalid entity classes and oversized subclasses.
- `OntologyBootstrapper { auth, ontology_store, app_data_dir }` with `pub async fn bootstrap(&self, sample_docs: &[BootstrapSampleDoc]) -> Result<Option<BootstrapSeed>, AppError>`:
  1. Idempotency check (`bootstrap_completed()` → `Ok(None)`)
  2. Provider check (`get_active_provider()` → `Ok(None)` if absent)
  3. Model selection via `Pass2LlmRefiner::pick_model_default`
  4. Format sample docs into one user message
  5. `AIServiceRequest` with `temperature=0.2`, `max_tokens=2048`
  6. `ai_request_with_retry(..., 3)`
  7. Parse + validate
  8. `apply_bootstrap` + `save` (save errors logged, not propagated)
  9. Returns `Ok(Some(seed))`
- 15 unit tests: JSON parsing (direct/fence-stripped/partial-defaults), validation (seed dups, bad names, dedup, cap headroom, bad entity class, valid subclass), bootstrapper idempotency (no-provider, already-completed), prompt content, sample formatting, RFC3339 helper.

**`src-tauri/src/pipeline/backfill.rs`** (Task 2 wiring):
- Line 116: `let mut pass2_success_count: u32 = 0;`
- Line 117-118: `bootstrap_samples: Vec<BootstrapSampleDoc>` capacity-reserved to `BOOTSTRAP_MIN_DOCS`
- Line 118-122: `OntologyBootstrapper::new(auth.clone(), ontology_store.clone(), app_data_dir.clone())` constructed once per backfill run
- Inside the per-doc loop's `Ok(entities_version)` arm: increments `pass2_success_count` when the doc reached `TWO_PASS_TARGET_VERSION` or `PASS3_TARGET_VERSION`; collects a `BootstrapSampleDoc` via `build_bootstrap_sample` while under 30 samples
- Line 196: `should_trigger_bootstrap(pass2_success_count, ontology_store.lock().await.bootstrap_completed())` gates the actual `bootstrapper.bootstrap(&bootstrap_samples).await` call; outcome logged via `eprintln!`; `bootstrap_samples` cleared + shrunk after firing (fires at most once per process, regardless of how many docs are processed after 30)
- Line 320: `fn should_trigger_bootstrap(count: u32, already_completed: bool) -> bool` — pure arithmetic gate, unit tested directly
- Line 328: `async fn build_bootstrap_sample(engine, doc_id) -> Option<BootstrapSampleDoc>` — reads `name`/`title`, `topic`, `llmTags`, and up to 5 `extracted_entities` (class, value) pairs from the doc's freshly-written vector-entry metadata; returns `None` on any lookup failure (tolerated by the caller)
- `spawn_entity_backfill` signature grew two new parameters: `ontology_store: Arc<tokio::sync::Mutex<OntologyStore>>` and `auth: Arc<AuthState>` (inserted after `triple_store`, before `embedder`)

## Deviation: AppState Wiring (Rule 3 — Blocking Issue)

The plan's `files_modified` scoped this plan to `backfill.rs` only, explicitly deferring `AppState`/`lib.rs` wiring to Plan 06 ("Plan 06 wires the actual construction"). However, changing `spawn_entity_backfill`'s signature broke compilation at its 3 existing call sites, and the plan's own `<verify>` block requires `cargo test --lib pipeline::backfill` to pass — which requires the whole crate to compile.

Rather than leave the crate in a broken state for a future plan to unblock, this plan applied the minimal fix (Rule 3):
- Added `ontology_store: Arc<Mutex<OntologyStore>>` and `auth_state: Arc<AuthState>` fields to `AppState` (`state.rs`)
- Constructed `OntologyStore::load(&app_data)` in `lib.rs` setup, mirroring the existing `triple_store` sidecar-load pattern
- Cloned `auth_arc` (already existed for `TwoPassExtractor::new`) before it moved, so it could also be stored in `AppState` and passed to the boot-time backfill call
- Updated all 3 call sites: `lib.rs` (boot-time backfill), `commands/folders.rs` (`trigger_scan` live-index backfill), `commands/entities.rs` (`trigger_entity_backfill` IPC command)

This is exactly the wiring Plan 06 would have needed to add for its own Settings > Ontology IPC commands (which also need `state.ontology_store`) — so Plan 06's scope is now narrower, not broader.
