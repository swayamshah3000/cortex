# Phase 11.6: Adaptive Ontology + Corpus-Seeded Bootstrap - Context

**Gathered:** 2026-07-09
**Status:** Ready for planning
**Trigger:** Real-corpus test — user wants cortex to auto-learn domain-specific predicates + entity subtypes + prominence rankings from their actual documents, not just apply the 21 hardcoded Phase 11.5 predicates.

<domain>
## Phase Boundary

Phase 11.6 makes cortex's ontology (predicates + entity classes + subtypes) **grow from the user's corpus** instead of being frozen at the 21-predicate + 8-class seed.

**Six deliverables:**

1. **Corpus-seeded bootstrap** — first-run background analysis of 30-50 sample docs to propose domain-specific extensions to the seed vocabulary (car → `registered_to`, `insured_by`, `taxed_by`; property → `apartment_of`, `plot_in`; person → `spouse_of`, `sibling_of`).

2. **Adaptive predicate discovery** — Pass 3 optionally emits `new_predicates: []` alongside triples. Predicates get promoted to the vocabulary after appearing in ≥ 2 different docs (dedup + minimum-support gate).

3. **Entity normalization** — post-Pass 2 normalizer producing `canonical_short_name` for verbose entities (`Alpha Beta Complex-Unit-204` → `Unit 204`; `Jane Q Doe` stays as-is because full name is already canonical).

4. **Frequency-weighted entity ranking** — Sidebar + entity pages sort by doc_count; top-N gets prominence (bold, larger card, first in list).

5. **Consolidation loop** — background task every 500 new triples OR 24h. LLM scans current vocabulary, proposes merges (`owned_by` + `belongs_to` → same), renames, splits. Persists as `pending_consolidation` in taxonomy store; user approves in Settings.

6. **Settings → Ontology panel** — table view of all predicates + entity classes + first-seen doc + count. Accept/reject consolidation diffs. Manual add/rename/merge. Reset to seed.

### Out of scope
- Multi-hop reasoning across triples (Phase 13-ish).
- Per-corpus rules (cortex learns globally, not per-folder).
- Prompt engineering UI — users can't edit LLM prompts (v1.3).
- Ontology import/export (v1.3 candidate).
- ML-based entity resolution — token-subset + cosine (Phase 6/11) stays.

</domain>

<decisions>
## Implementation Decisions

### Corpus-Seeded Bootstrap (D-01..D-04)

- **D-01:** Fires ONCE per install after first backfill batch reaches 30 docs. Reads their titles + topics + tags + top-5 entities into a single LLM call: *"Given these 30 documents from a personal document library, propose 20-30 domain-specific predicates and entity subclasses that would be useful to extract. Return: `{predicates: [{name, description, example_use}], entity_subclasses: [{class, subclass, example}]}`."*
- **D-02:** Bootstrap runs on Anthropic Haiku default. Costs ~$0.005 per bootstrap. Result cached to `app_data_dir/ontology.json` — never re-runs unless user clicks "Regenerate ontology" in Settings.
- **D-03:** Bootstrap output = SEED additions. Merges into the 21-baked-in Phase 11.5 predicate list. Baked-ins survive user reset; corpus-seeded ones are removable.
- **D-04:** Skip bootstrap if user has < 30 docs. Cortex just uses the 21 seed predicates until the threshold is hit.

### Adaptive Predicate Discovery (D-05..D-08)

- **D-05:** Pass 3 prompt extended: after emitting triples with baked predicates, LLM MAY emit `new_predicates: [{name, description, subject_class, object_class}]` for relations not fitting existing vocabulary. Max 3 per doc.
- **D-06:** New predicates land in `pending_predicates` — not immediately promoted. Promoted after appearing in ≥ 2 different docs (minimum support gate). Prevents one-off hallucinations from polluting vocab.
- **D-07:** Promoted predicates persist to `app_data_dir/ontology.json` alongside bootstrap seeds. Get full vocab treatment in downstream Pass 3 calls (LLM can reuse them like baked-ins).
- **D-08:** Vocabulary hard cap = 200 predicates. When cap hit, consolidation loop must run before more can be added. Prevents runaway growth.

### Entity Normalization (D-09..D-11)

- **D-09:** New Rust module `pipeline/entity_normalizer.rs`. Runs after Pass 2 merge (before entity_store registration). Takes each entity's raw text + class + subclass → returns `canonical_short_name: String`.
- **D-10:** Rule-based: strip common corporate suffixes (Ltd, Inc, Corp); collapse whitespace; if class=Location AND name contains multiple hyphens/dashes, keep the last segment ONLY if it looks like a unit number (regex: `\w+-?\d+$`). Falls back to raw text.
- **D-11:** `EntityStore.canonical_name` uses `canonical_short_name` if present, else raw. UI reads canonical_short_name for chip display; raw stays as first alias.

### Frequency-Weighted Entity Ranking (D-12..D-14)

- **D-12:** Every entity page shows entities sorted by `doc_count DESC`. Top-3 get `is_prominent: true` flag added to the response type.
- **D-13:** Sidebar Entities section shows top-5 prominent entities inline (below Saved Searches). Click → entity detail page.
- **D-14:** Sidebar "Owned by me" quick link keeps auto-detection but uses new prominence signal — top Person by doc_count.

### Consolidation Loop (D-15..D-17)

- **D-15:** Triggers every 500 upserted triples OR every 24h whichever first. Runs in background via `tokio::spawn`. Skips if user hasn't opted in via Settings toggle.
- **D-16:** LLM receives current predicate list + entity subclass list. Prompt: *"Which of these predicates are near-synonyms and should merge? Which are overloaded and should split? Return `{merges: [[a, b] → c], renames: [[old, new]], splits: [old → [a, b]]}`. Only propose changes you're confident on."*
- **D-17:** Result stored as `pending_consolidation` in `ontology.json`. Frontend Settings → Ontology → pending diffs panel. Applied atomically on user approval — re-tags affected triples in TripleStore.

### Ontology Store (D-18..D-20)

- **D-18:** New `graph/ontology_store.rs` module. Schema:
  ```json
  {
    "version": 1,
    "seed_predicates": [...21 baked-ins...],
    "corpus_seed": [...bootstrap output...],
    "adaptive_predicates": [{ name, description, count, first_seen, promoted_at }],
    "pending_predicates": [{ name, description, count, first_seen }],
    "entity_subclasses": { "Location": ["apartment", "plot", ...], ... },
    "pending_consolidation": { merges, renames, splits, generated_at }
  }
  ```
- **D-19:** JSON sidecar in `app_data_dir/ontology.json`. Mirrors `SpaceLabelCache` / `SavedSearchStore` / `TripleStore` pattern. Loaded once at boot into AppState.
- **D-20:** Consolidation state cleared when user accepts/rejects. Both actions log to `activity_log`.

### Settings → Ontology Panel (D-21..D-23)

- **D-21:** New tab in Settings between "AI Providers" and "Extraction". Sections:
  - Predicates table (name, count, source [seed/corpus/adaptive/manual], first-seen doc)
  - Entity subclasses table
  - Pending consolidation diff (accept/reject per suggestion)
  - Manual actions: Add predicate, Rename, Merge two, Delete adaptive, "Reset to seed"
  - Enable toggle: "Automatic ontology growth" (opt-in default OFF for privacy-strict users)
- **D-22:** Reuse Phase 11.5 UI patterns (tables, chips, AlertDialog for destructive actions).
- **D-23:** Sidebar quick link "Ontology" surfaces the pending count as a badge when consolidation is pending.

### Claude's Discretion (Planner-owned)

- IPC command names (planner finalizes): `get_ontology`, `apply_consolidation`, `add_manual_predicate`, `rename_predicate`, `merge_predicates`, `reset_ontology_to_seed`, `regenerate_corpus_seed`.
- React Query key strategy.
- Whether to run corpus-seed bootstrap synchronously (blocks first-doc extraction) or async (Pass 3 uses fixed seed until bootstrap completes).
- Entity normalizer exact regex set — planner may add more heuristics from corpus samples.
- Whether to show consolidation diff as unified list or per-predicate cards.
- Bootstrap prompt engineering — planner iterates against user's ~/private sample.

<specifics>
## Specific Ideas

- **Corpus-seed bootstrap is the killer feature** — cortex "wakes up smart" for the user's domain instead of trying to auto-discover from zero. Sample ~/private/docs shows: `owner_of`, `neighboring_plot_of`, `sale_deed_of`, `mortgaged_to`, `tax_receipt_for`. Cortex should propose these on install.
- **Family relations from Indian ID docs** — passport has spouse + children; PAN has father's name; Aadhaar has address links. Phase 11.5 has `married_to, parent_of, child_of` in seed vocab. Bootstrap likely adds `father_of, mother_of, sibling_of` from surname patterns.
- **Property normalization concrete example** — user has `Alpha Beta Complex-Unit-204`, `Complex-Unit`, `Unit 204` variants across docs. Normalizer rule: last hyphen-segment ending in digits = canonical short. Rare special cases (Riverside Complex P705) stay full.
- **Frequency ranking = prominence** — Alex Doe in 500 docs is more important than Rahul Banerjee in 1 doc. Sidebar top-5 by count. Entity page footer shows "Top 10 people in your library" ranked by count.
- **Bootstrap is opt-in reveal** — Settings > Ontology shows "cortex found 24 domain-specific patterns in your library. Accept?" on first visit after bootstrap completes.

</specifics>
