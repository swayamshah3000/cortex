# Phase 11.6 Plan 05: Pass 3 Dynamic Vocabulary Summary

**Pass 3 relation extractor now builds its prompt from `OntologyStore.effective_predicate_names()` at call time instead of a hardcoded 21-predicate const, and can propose up to 3 new predicates per doc that feed back into the ontology's min-support promotion gate.**

## Performance

- **Duration:** ~55 min
- **Tasks:** 2 completed
- **Files modified:** 6

## Accomplishments
- `build_extract_relations_prompt(vocab: &[String]) -> String` replaces the fixed `EXTRACT_RELATIONS_PROMPT` const, injecting the runtime vocabulary into RULES §2 while keeping rules 1/4-8, the JSON schema, and all 4 few-shot examples byte-identical to Phase 11.5
- `Pass3RelationExtractor.extract_full()` returns `(Vec<Triple>, Vec<Pass3NewPredicate>)` given a runtime vocabulary; `extract()` becomes a thin backward-compat wrapper over `extract_full()` using `SEED_PREDICATES`
- `validate_and_normalize()` now takes the runtime vocabulary as a parameter and validates `new_predicates`: snake_case + length cap, no duplicate-with-vocabulary, batch dedup, capped at 3/doc (cap applied last, after filtering)
- `backfill.rs` fetches `OntologyStore.effective_predicate_names()` before every Pass 3 call and feeds `new_predicates` back via a new `feed_new_predicates_to_ontology()` helper, gated by `automatic_growth_enabled` (D-21)
- `OntologyStore` is now fully wired end-to-end (state.rs, lib.rs, both backfill trigger sites) and persists to `ontology.json` at the end of every backfill run

## Decisions Made
- Split `EXTRACT_RELATIONS_PROMPT` into three consts (`_PREFIX`, `_VOCAB_TEMPLATE`, `_SUFFIX`) rather than a single templated string, so the few-shot examples and static rules live in one immutable literal that can never accidentally diverge from Phase 11.5's wording — only the small vocab-lock rule (§2/§3) is templated.
- Kept a `#[deprecated]` `extract_relations_prompt_seed_only()` function (not a re-exposed `EXTRACT_RELATIONS_PROMPT` const, since the plan's builder pattern makes a static const impossible) so the legacy `test_extract_relations_prompt_contains_all_predicates` test still compiles and exercises the same assertion via the new builder.
- `feed_new_predicates_to_ontology` is `async fn` taking `&Arc<Mutex<OntologyStore>>` (not a sync helper) since it must hold the lock across `record_pending_predicate` calls — mirrors the existing `os.lock().await` pattern already used for `effective_predicate_names()` in the same function.
