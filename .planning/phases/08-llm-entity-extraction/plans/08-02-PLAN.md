  <files>src-tauri/src/pipeline/pass1_pattern_extractor.rs</files>
  <behavior>
    - Aadhaar: `validate_aadhaar("234123412346")` uses `verhoeff::verify(&s)` — returns true for a real Aadhaar checksum-valid sample, false for `"234123412340"` (wrong last digit). Test with two known-good and two known-bad 12-digit strings.
    - IBAN: `validate_iban("GB29 NWBK 6016 1331 9268 19")` returns true; `"GB29 NWBK 6016 1331 9268 20"` returns false; spaces are stripped before parse (Pattern 5 in RESEARCH.md).
    - Credit card: `validate_credit_card("4532015112830366", "context")` — Luhn passes AND Visa BIN "4" prefix → true; `"4532015112830367"` → false (Luhn fails); Luhn passes with unknown BIN AND no context word ("card"/"visa"/"mastercard"/"amex") → false (per D-03, "Luhn alone is 10% false-positive"); Luhn passes with unknown BIN AND context word "credit card" in the surrounding text → true.
    - BIN prefix helper: matches Visa (starts with 4), Mastercard (51-55 or 2221-2720), Amex (34 or 37), Discover (6011 or 65 or 644-649).
    - PAN: `validate_pan("ABCDE1234F")` → true; `"ABCDE12345"` (no trailing letter) → false; `"abcde1234f"` (lowercase) → false (D-03 requires strict uppercase).
    - SSN: `validate_ssn("123-45-6789")` → true; `"000-45-6789"` (invalid area code) → false; area code must not be 000, 666, or 900-999 per D-03 "area-code sanity".
    - NINO: `validate_nino("AB123456C")` → true; `"AA123456C"` (AA disallowed per regex `^[A-CEGHJ-PR-TW-Z]{2}...`) → false.
    - SIN: `validate_sin("046454286")` → true (Luhn-valid); `"046454287"` → false.
    - VIN: 17-character VIN with the NHTSA weighted checksum digit at position 9 — implement per RESEARCH.md Open Question 3 (~30 LoC). `validate_vin("1M8GDM9AXKP042788")` → true (this is a documented NHTSA test-VIN); a corrupted variant returns false. Include weighted table constants as `const VIN_WEIGHTS: [u32; 17]` and letter-to-value map.
    - GSTIN: 15-character format `[A-Z0-9]{15}` + custom Mod-36 Luhn-variant checksum (Open Question 1, ~30 LoC). Test with two known-good GSTIN samples (e.g., `"22AAAAA0000A1Z5"` — verify checksum matches the last char).
    - Weak-format fallback (D-04): text like "Policy number: POL-2024-4531" produces an ExtractedEntity with `entity_type="identifier"`, `class=Some("Identifier")`, `subclass=Some("unknown")`, `confidence=Some(0.7)`. The Pass 1 extractor SHOULD detect these via a regex that matches "policy number", "folio number", "account number", "receipt number", "invoice number", "plot number" followed by an alphanumeric token, and emit the alphanumeric part as the value.
  </behavior>
