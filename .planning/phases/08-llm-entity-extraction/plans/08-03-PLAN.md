<objective>
Build the LLM refinement pass: given document text + title + Pass 1 findings, call the active AI provider's chat endpoint once with the fixed 8-class schema prompt and return structured JSON (additional_entities, refined_entities, topic, tags). This is the LLM-driven half of the two-pass engine; when no provider is connected, refine() short-circuits to an empty output — Pass 1 alone still runs.

Purpose: Delivers LLME-01 (LLM extracts Person/Org/Location/topic/tags) and LLME-02 (uniform 8-class coverage across any domain). Enforces LLME-03 idempotence via `temperature = 0` + pinned model + versioned prompt.

Output: `pipeline/pass2_llm_refiner.rs` with `Pass2LlmRefiner::new(auth: Arc<AuthState>)`, `pub async fn refine(text: &str, title: &str, pass1: &[ExtractedEntity]) -> Result<Pass2Output, AppError>`, the inline REFINE_PROMPT constant with multi-region few-shot examples, and unit tests using a mock ai_request wrapper.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| LLM provider response → parse | Untrusted JSON string from Anthropic/OpenAI/Gemini/Ollama enters Rust via ai_request |
| Document text → prompt | Document content is inlined into the user message and sent to a cloud LLM |
| pass1_entities → prompt | Pass 1 output (which may include OCR-corrupted amounts, phone numbers) is inlined into the prompt |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-08-08 | Tampering | LLM JSON injection — adversarial doc content influences JSON shape | mitigate | `normalize_and_validate` enforces the 8-class allow-list; unknown classes dropped; `serde(default)` on all optional fields so partial responses don't panic |
| T-08-09 | Information Disclosure | Document content sent to cloud LLM | mitigate | D-33 "Use LLM for entity extraction" toggle in Settings; when off, Pass 2 is skipped; refine() returns empty when no provider is connected (D-31); user visibility via AiNoProviderBanner |
| T-08-10 | Denial of Service | Semaphore exhaustion (all 8 permits stuck) | accept | Per RESEARCH.md Security Domain: rely on reqwest's ~30s built-in timeout to unblock stuck permits; explicit permit timeout deferred to v1.2 |
| T-08-11 | Denial of Service | LLM emits multi-megabyte response, exhausts memory | mitigate | max_tokens=4096 in AIServiceRequest — bounds provider output; provider-side enforcement (all four providers enforce max_tokens) |
| T-08-12 | Denial of Service | Unbounded concurrency into Anthropic/OpenAI RPM | mitigate | tokio Semaphore cap=8 (D-13); respects 1000 RPM for Anthropic prod |
</threat_model>
