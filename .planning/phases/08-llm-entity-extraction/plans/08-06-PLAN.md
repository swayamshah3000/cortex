<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| user click "Re-extract" → backfill task | Long-running task spawns with mutable access to EntityStore + engine + LLM provider |
| provider disconnect mid-backfill → task | Provider handle can vanish while the task is iterating |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-08-18 | Denial of Service | Repeated Re-extract clicks spawn N parallel backfill tasks | mitigate | Frontend button disables while backfill is running (Plan 07 owns the disabled state via `useBackfillStore`); backend does not enforce single-flight — accept small window of double-spawn as they'll idempotently gate on `entities_version >= 3.0` |
| T-08-19 | Denial of Service | Backfill hangs after provider disconnect | mitigate | Pitfall 7 mitigation: per-doc provider-connection check inside the loop; in-flight LLM call still bounded by reqwest ~30s timeout inherited from Phase 7 |
| T-08-20 | Tampering | Backfill leaves partial metadata on crash | accept | Each doc is a separate metadata insert; a partial run leaves some docs at 2.5 and some at 3.0 — the next Re-extract completes the rest. Idempotent by design (LLME-03) |
</threat_model>
