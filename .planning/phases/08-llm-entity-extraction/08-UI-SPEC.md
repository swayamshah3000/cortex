## Visual Hierarchy

**Primary focal point of the ExtractionSettings section (Settings â†’ AI tab):** the "Re-extract entities" button. It is the only control rendered in `btn-primary` (accent-fill) style; every other control in the section (model dropdown, LLM toggle) is a neutral input whose only job is to gate the button's availability state. Executors implementing the section must preserve this hierarchy: dropdown and toggle sit above with neutral chrome, and the accent-filled button anchors the section as the action the user came here to take.

**Primary focal point of the /document/:id metadata sidebar (phase 8 additions):** the Topic chip (accent-tinted `TopicChip`) rendered as the first item in the entity section. It is the doc-level semantic label; tag chips and per-entity chips follow with progressively lower visual weight. When no topic is assigned, the first EntityChip becomes the focal point by position.

**Primary focal point of the TopicFilterBar (on /search and /tags):** the row of chips itself is uniform (no single chip is highlighted at rest); an active-state chip (accent-fill) becomes the focal point once selected. This mirrors the existing `FilterChip` pattern in `SearchPage.tsx`.

**Icon-only actions:** phase 8 does not introduce any icon-only interactive controls. Every interactive element carries either a text label, an aria-label, or a tooltip. The `ConfidenceExpander` trigger uses ChevronDown + "Also found ({count})" text. Dismiss buttons reuse the existing 44 Ă— 44 pattern with aria-label from `AiNoProviderBanner`.

## Topic vs Tag Visual Distinction Contract

This is the critical semantic differentiation in phase 8. Executors must NOT use the same chip style for both.

| Dimension | Topic chip | Tag chip |
|-----------|-----------|----------|
| Semantic | Single-value, doc-level category (from 19-topic seed or free-form LLM) | Multi-value, free-form keywords (2-5 per doc, LLM-generated) |
| Icon | `Bookmark` (12px) | `Hash` (12px) |
| Background | `bg-accent-primary/10` (indigo tint) | `bg-bg-tertiary` (neutral dark) |
| Text color | `text-accent-primary` | `text-text-secondary` |
| Border | `border-accent-primary/20` | `border-border-secondary` |
| Font | `text-xs` (weight 400) | `text-xs` (weight 400) |
| Border radius | `rounded-full` (pill) | `rounded-md` (rectangular pill) |
| Padding | `px-3 py-1` (12px / 4px, grid-aligned) | `px-3 py-1` (12px / 4px, grid-aligned) |
| Max width | none (single value, typically short) | `max-w-[120px] truncate` |
| Display text | `_` â†’ space, capitalize first letter: `term_insurance` â†’ "Term insurance" | `_` â†’ space: `khush_school` â†’ "khush school" |
| Clickable (phase 8) | No (`div`, not `button` or `Link`) | No (`div`, not `button` or `Link`) |
| Phase 11 upgrade | Becomes filter link | Becomes filter link |

**In the `/tags` page tag cloud:** Only free-form `tags` are shown. The `topic` field is handled by `TopicFilterBar` separately. The two MUST NOT appear in the same cloud.

**Visual weight distinction:** Topic chip is the higher-weight signal (accent-tinted background). Tag chip is intentionally lower-weight (neutral background). This asymmetry â€” reinforced by pill shape (topic) vs rectangular pill (tag) â€” ensures the eye reads doc-level topic before scanning multi-value tags, matching semantic priority.
