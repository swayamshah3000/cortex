<objective>
Port the `ai/` module from learnforge into Cortex as the provider routing layer. This delivers the central `ai_request()` router that consumes `AuthState` (from Plan 01), picks a provider branch via `normalize_provider_name`, and dispatches to per-provider chat functions for Anthropic, OpenAI, Gemini, and Ollama. After this plan: `cargo test -p cortex_lib ai` runs the full ported test suite (13+ tests) green, and the backend has every piece needed to send a real LLM request once IPC wiring lands in Plan 03.

Purpose: The `ai/` module is the second half of the backend port. It has the most complex per-provider request shapes (Anthropic system-as-top-level vs. OpenAI system-as-message — Pitfall 4), the OAuth setup-token header construction (Pitfall 3), and the retry/backoff utility that Phase 8's high-volume entity extraction will depend on. Porting verbatim preserves the test coverage that proves each per-provider request shape is correct.

Output: 5 new Rust files under `src-tauri/src/ai/`, one new `mod ai;` declaration in lib.rs. No IPC commands registered (Plan 03 owns invoke_handler). Plan 01 already added `mod auth;` and reqwest dependency, so `ai/` compiles directly against the existing crate.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Tauri process → Anthropic / OpenAI / Gemini API | All chat requests cross HTTPS to provider APIs |
| Tauri process → Ollama (local network) | `ollama_chat` posts to user-configured `base_url` (default `http://localhost:11434`) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-06 | Information Disclosure | Token leak in error body via `map_anthropic_error` / `map_openai_error` | mitigate | Error mappers truncate body to 200 chars; tokens are NEVER substituted into error strings (they're sent in headers, not the response body). Verified by reviewing learnforge mapper functions. |
| T-07-07 | Elevation of Privilege | SSRF via Ollama `base_url` | mitigate | `ollama_chat` reads `base_url` from `ProviderCredential.base_url` which is set only via `store_ollama_config` IPC (Plan 03). Validation of scheme `http://` or `https://` is enforced at the UI level (Plan 05); backend trusts what the credential store delivers. No redirect following enabled in reqwest default client. |
| T-07-08 | Spoofing | Forged Anthropic `anthropic-beta` header acceptance | accept | Anthropic API enforces this header server-side; Cortex sends what learnforge sends. If Anthropic deprecates `oauth-2025-04-20` (Assumption A1), validation will fail loudly with 401 — fix forward by updating the header constant. |
| T-07-09 | Denial of Service | Runaway retry on persistent 429 | mitigate | `retry_with_backoff` caps at `max_retries` (Plan 03 uses 2); exponential doubling from 2s means max wait is 2s + 4s = 6s for max_retries=2. No infinite loop possible. |
</threat_model>
