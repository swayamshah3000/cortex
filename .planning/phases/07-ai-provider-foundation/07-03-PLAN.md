<objective>
Build the IPC bridge that lets the frontend call into the `auth/` and `ai/` modules from Plans 01-02. This plan creates `commands/ai.rs` (8 thin wrapper commands), declares it in `commands/mod.rs`, and updates `lib.rs` to (a) call `app.manage(auth_state)` and `app.manage(oauth_flow_state)` BEFORE `app.manage(AppState {...})` (Pitfall 1 — wrong order causes Tauri panic), and (b) register all 8 commands in `invoke_handler`. After this plan: a `bun run tauri dev` instance accepts Tauri invokes for `list_providers`, `connect_provider`, `save_setup_token`, etc., and `chat` round-trips through `ai_request_with_retry` to a real provider when credentials are configured.

Purpose: Plans 01-02 created pure Rust modules with no Tauri attribute exposure. The frontend cannot call those functions until they're registered as `#[tauri::command]` and wired into `invoke_handler`. This plan delivers that bridge. It also exposes `chat()` in Phase 7 (RESEARCH Open Question #1 recommendation) so the planner can manually UAT a real Anthropic round-trip from DevTools before Phase 8 ships.

Output: 1 new file `commands/ai.rs` (~100 lines), 1-line addition to `commands/mod.rs`, 2 logical edits to `lib.rs` (manage() ordering + invoke_handler additions). No new frontend code (Plan 04 owns hooks and types).
</objective>

    2. `connect_provider(auth: State<'_, AuthState>, request: LoginRequest) -> Result<ProviderAuthStatus, String>` — async; **CRITICAL D-08 WIRING — validate before store**:
       - Match on `request.method.as_str()`:
         - `"api-key"`: Match on `request.provider.as_str()`:
           - `"openai"` → call `crate::auth::oauth::validate_openai_token(&credential).await?` BEFORE calling login_provider
           - `"gemini"` → call `crate::auth::oauth::validate_gemini_token(&credential).await?` BEFORE calling login_provider
           - other providers with method="api-key" → return `Err("API-key auth not supported for this provider".to_string())` (Anthropic uses save_setup_token; OpenAI/Gemini are the only API-key providers in v1.1)
         - `"ollama"`: call `crate::auth::oauth::validate_ollama_endpoint(&base_url).await?` BEFORE calling login_provider (where base_url comes from `request.base_url.as_deref().unwrap_or("http://localhost:11434")`)
         - other → return `Err(format!("Unsupported auth method: {}", request.method))`
       - On validation error: map via `crate::auth::oauth::map_oauth_error(&e)` and return `Err(mapped_msg)`. NEVER call `login_provider` if validation fails — credential must not enter the store.
       - On validation success: delegate to `crate::auth::commands::login_provider(auth, request)` which performs the actual store + return `ProviderAuthStatus`.

       Pseudocode shape (full impl in actual Rust):
       ```
       match request.method.as_str() {
           "api-key" => {
               let credential = request.credential.as_deref().ok_or("API key is required")?;
               match request.provider.as_str() {
                   "openai" => validate_openai_token(credential).await.map_err(|e| map_oauth_error(&e))?,
                   "gemini" => validate_gemini_token(credential).await.map_err(|e| map_oauth_error(&e))?,
                   p => return Err(format!("API-key auth not supported for provider: {}", p)),
               }
           }
           "ollama" => {
               let base_url = request.base_url.as_deref().unwrap_or("http://localhost:11434");
               validate_ollama_endpoint(base_url).await.map_err(|e| map_oauth_error(&e))?;
           }
           other => return Err(format!("Unsupported auth method: {}", other)),
       }
       // Validation passed — now store.
       crate::auth::commands::login_provider(auth, request)
       ```
