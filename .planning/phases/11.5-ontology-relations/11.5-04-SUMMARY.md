# Phase 11.5 Plan 04: Backfill Wiring Summary

**Wires Pass 3 relation extraction into the existing backfill loop as a Pass-2-dependent stage — TwoPassExtractor gains a pass3() accessor, backfill_one_doc_async upserts Pass 3 triples into TripleStore with doc provenance and bumps entities_version 3.0 → 3.5, and collect_backfill_candidates re-gates on PASS3_TARGET_VERSION so existing v3.0 docs are automatically re-processed.**

## Accomplishments

- `TwoPassExtractor` now holds `pass3: Arc<Pass3RelationExtractor>`, constructed alongside Pass 2 in `new()`; `pass3()` accessor added mirroring the existing `pass2()` pattern
- `set_model()` propagates the model to both Pass 2 and Pass 3 (D-07: same active provider + model, no new Settings surface)
- `extract_full()` doc comment clarifies Pass 3 runs separately in the backfill worker, keeping the fast indexer path (Pass 1 + Pass 2 only) unchanged
- `backfill_one_doc_async` runs Pass 3 immediately after `register_doc_entities` populates canonical_ids, gated on `entities_version == TWO_PASS_TARGET_VERSION` (Pass 2 succeeded): calls `two_pass.pass3().extract(text, title, &entities_vec)`, then on `Ok(Some(triples))` runs `cleanup_doc(doc_id)` followed by `upsert_from_doc(doc_id, triples)` inside a `spawn_blocking` block holding the TripleStore mutex
- On Pass 3 success, `final_version` advances to `PASS3_TARGET_VERSION` (3.5); on upsert failure, LLM error, or `Ok(None)` (no provider / no entities), `final_version` stays at 3.0 so the doc is retried on the next backfill run
- `collect_backfill_candidates` gate changed from `< TWO_PASS_TARGET_VERSION (3.0)` to `< PASS3_TARGET_VERSION (3.5)`; the early-return gate in `backfill_one_doc_async` and the "doc disappeared" fallback both updated to match
- `spawn_entity_backfill` persists `TripleStore` to disk once per backfill session (after the alias-merge step, before the final "complete" event) via `spawn_blocking` + `blocking_lock` + `TripleStore::save`
- **Compile-fix ripple (Rule 3):** `spawn_entity_backfill`'s new `triple_store` + `app_data_dir` parameters broke all 3 existing call sites. Added `AppState.triple_store: Arc<tokio::sync::Mutex<TripleStore>>`, loaded via `TripleStore::load(&app_data)` in `lib.rs` setup (mirrors the `SavedSearchStore`/`SpaceLabelCache` sidecar-load pattern already there), and threaded `state.triple_store.clone()` + `state.app_data_dir.clone()` through `commands/entities.rs::trigger_entity_backfill`, `commands/folders.rs::trigger_scan`, and the boot-time backfill call in `lib.rs`
- Added 5 new unit tests: `test_pass3_accessor_returns_arc`, `test_set_model_propagates_to_pass3` (two_pass_extractor.rs); `test_collect_backfill_candidates_picks_up_v3`, `test_collect_backfill_candidates_excludes_v35`, `test_collect_backfill_candidates_gate_v2_v25_v3_v35_coverage` (backfill.rs)
- Updated 2 pre-existing tests (`test_collect_backfill_candidates_excludes_v3`, `test_collect_backfill_candidates_gate_coverage`) whose assertions were obsoleted by the gate change

## Decisions Made

- **AppState.triple_store added now, not deferred to Plan 05.** The plan's interface section says "AppState will grow a `triple_store` field ... that wiring is Plan 05's job. Here we take it as a parameter ... so this plan does not touch state.rs." However, changing `spawn_entity_backfill`'s signature is a hard compile break at 3 call sites the plan didn't fully enumerate (`lib.rs` boot-time backfill and `commands/folders.rs::trigger_scan`, in addition to the plan-cited `commands/entities.rs::trigger_entity_backfill`). Per Rule 3 (auto-fix blocking issues — "missing referenced file", "build config error" class), the crate must compile after this plan lands. Rather than stub a throwaway `TripleStore::default()` at each call site (which would silently discard triples across backfill runs), a real `AppState.triple_store` field loaded from `triples.json` was added — functionally correct, and it directly becomes the value Plan 05 was going to add anyway. Plan 05's own scope (documented as "must add `state.triple_store.clone()` and `state.app_data_dir.clone()` to this call") is now already satisfied for `trigger_entity_backfill`; Plan 05 should verify this rather than redo it.
- **Test correction, not test deletion.** `test_collect_backfill_candidates_excludes_v3` and `test_collect_backfill_candidates_gate_coverage` asserted "v3.0 must NOT be a candidate" — true under the old 3.0 gate, false under the new 3.5 gate per this plan's own must-haves ("collect_backfill_candidates now picks up docs with entities_version < PASS3_TARGET_VERSION"). Assertions were flipped to match the new correct behavior; test names and fixtures kept for historical continuity, with comments pointing to the newer Phase-11.5-authored equivalent tests that cover the same ground.
- **Pass 3 stage placement:** after `register_doc_entities` (so `entities_vec` has `canonical_id`s populated — required by `Pass3RelationExtractor::extract`'s canonical-id mapping step) and before the metadata-write block, so `final_version` is fully resolved before it's serialized into `entities_version`.
