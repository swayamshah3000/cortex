# Phase 11.5: Ontology / Relation Extraction - Context

**Gathered:** 2026-07-08
**Status:** Ready for planning
**Trigger:** User real-corpus test — entities extracted but no cross-doc relationships. "All my assets owned by X" doesn't work.

<domain>
## Phase Boundary

Phase 11.5 adds the **relation extraction layer** — the missing "knowledge graph" that connects entities across documents. After Pass 2 extracts entities per doc, a new **Pass 3 (Relation Extractor)** calls the LLM once more to identify triples `(subject_entity, predicate, object_entity)`. These triples land in a persisted `TripleStore` queryable via IPC + UI.

Enables:
- "All documents related to Unit 204" (via `?doc mentions Raga_2004` OR `?owner --owns--> Raga_2004`)
- "All assets owned by Alex Doe" (via `Alex_Shah --owns--> ?`)
- "Who is Jane related to?" (via `?person --related_to--> Jane`)
- Entity detail page "Relations" panel — shows `(this entity, predicate, other entity)` triples

### What Phase 11.5 delivers

1. **Pass 3 Relation Extractor** — new `pipeline/pass3_relation_extractor.rs`. Runs during backfill AFTER Pass 2 completes. Takes doc text + Pass 2 entities → returns `Vec<Triple>`.
2. **Fixed predicate vocabulary (v1)** — 15 relations LLM must pick from. Prevents predicate explosion. Extensible via user-editable Settings panel (v1.2).
3. **TripleStore** — `graph/triple_store.rs`. In-memory + persisted to `app_data_dir/triples.json`. HashMap-backed for fast subject→objects + reverse object→subjects lookups.
4. **7 IPC commands** — `get_entity_relations`, `get_subjects_by_predicate_object`, `get_objects_by_subject_predicate`, `get_all_owned_by`, `get_all_related_to`, `delete_triple` (user override), `add_manual_triple` (user override).
5. **UI extensions** —
   - Entity detail page: new **Relations panel** below Documents section
   - New `/ownership/:person_id` route — grouped by asset type (Property / Vehicle / Investment / Business)
   - "Related to" section on Person entity pages
6. **Backfill re-wire** — Pass 3 runs on every backfill; existing docs re-processed on next "Re-extract entities" click.

### Out of scope

- User-editable predicate vocabulary (v1.2)
- Multi-hop reasoning ("friend of a friend")
- Temporal reasoning ("owned in 2020 but sold in 2024")
- Cross-provider relation extraction (uses active provider only)
- Automatic ontology learning (predicate set is fixed)

</domain>

<decisions>
## Implementation Decisions

### Relation Vocabulary (Fixed 15-predicate set)

- **D-01: Predicate closed set (v1):**
  - **Ownership:** `owns`, `owned_by` (reverse), `purchased_from`, `sold_to`
  - **Location:** `located_in`, `part_of`, `address_of`
  - **Personal:** `married_to`, `parent_of`, `child_of`, `member_of`
  - **Business:** `employer_of`, `employee_of`, `partner_of`
  - **Legal:** `issued_by`, `dated`, `signed_by`
  - **Identity:** `uses_pan`, `uses_aadhaar`, `has_voter_id`
- **D-02: LLM cannot invent new predicates.** Prompt states this explicitly. If a relation doesn't fit, LLM emits `mentioned_with` (weak default) or omits.
- **D-03: Relations are directional.** `A --owns--> B` is stored + auto-inverted to `B --owned_by--> A` at write time.

### Pass 3 Extraction (Area 2)

- **D-04: Runs after Pass 2 in backfill.** Skipped when no provider connected (like Pass 2). `entities_version = 3.5` = Pass 1 + Pass 2 + Pass 3 complete. `3.0` = Pass 1+2 only, will be upgraded next backfill.
- **D-05: LLM prompt takes Pass 2 entities + doc text.** Structure:
  ```
  Entities in this doc: [e_0: Person "Alex Doe", e_1: Location "Unit 204", ...]
  Doc text (heads+tails 6k): ...
  Output: [{subjectId: "e_0", predicate: "owns", objectId: "e_1"}, ...]
  ```
- **D-06: Reuse Phase 8 fence-strip + JSON parse.** Same robust patterns.
- **D-07: Same active provider + model as Pass 2.** No new settings.
- **D-08: Concurrency = 8 in-flight (matches Pass 2 semaphore).**

### TripleStore (Area 3)

- **D-09: In-memory HashMap-backed store.** Persisted to `app_data_dir/triples.json`.
- **D-10: Two indices:**
  - Forward: `HashMap<(subject_id, predicate), HashSet<object_id>>` — fast "X owns ?"
  - Reverse: `HashMap<(predicate, object_id), HashSet<subject_id>>` — fast "? owns Y"
- **D-11: Doc provenance per triple** — each triple carries `Vec<doc_id>` (which docs support it). Enables "show me the sale deed that proves this" UX.
- **D-12: User override support** — `add_manual_triple` + `delete_triple` IPCs. Manual triples carry a `user_added: true` flag and are preserved across LLM re-runs.

### IPC Surface (Area 4)

- **D-13:** `get_entity_relations(entity_id) → Vec<Triple>` — all triples touching this entity (subject or object).
- **D-14:** `get_all_owned_by(person_id) → HashMap<AssetType, Vec<Entity>>` — grouped by asset type derived from object entity's class + topic.
- **D-15:** `get_all_related_to(entity_id) → Vec<(predicate, Entity)>` — flat list.
- **D-16:** `add_manual_triple(subject, predicate, object, doc_id?)` + `delete_triple(triple_id)` — user override.

### UI (Area 5)

- **D-17: Entity detail page** — new **Relations section** below Documents. Rows: `→ predicate → target_entity_chip` + provenance doc count. Click target chip → nav to target entity page.
- **D-18: `/ownership/:person_id` new route** — grouped view:
  - **Property** (Location + topic=property/real_estate)
  - **Vehicle** (Location + topic=vehicle)
  - **Investment** (any entity + topic=investment)
  - **Business** (Organization + topic=business)
  - **Financial accounts** (Identifier + subclass matching bank_account/folio/policy)
  Renders as sections w/ SpaceCard-style entity cards.
- **D-19: Sidebar quick link** — "Owned by me" — nav to `/ownership/{first_person_by_doc_count}` (usually the user). One-click access.

### Backfill Integration (Area 6)

- **D-20: Pass 3 runs after Pass 2 in same backfill worker.** Backfill loop:
  ```
  for doc in candidates {
      pass2_output = pass2.refine(...)
      if pass2_output.is_some() {
          pass3_triples = pass3.extract(text, pass2_output.entities)
          triple_store.upsert_from_doc(doc_id, pass3_triples)
          set entities_version = 3.5
      } else {
          set entities_version = 2.5  // Pass 2 failed, Pass 3 skipped
      }
  }
  ```
- **D-21: `entities_version` semantics:**
  - `2` = BERT legacy
  - `2.5` = Pass 1 only (LLM unavailable)
  - `3` = Pass 1 + Pass 2 (existing v1.1 state)
  - `3.5` = Pass 1 + Pass 2 + Pass 3 (Phase 11.5 target)
- **D-22: TWO_PASS_TARGET_VERSION bumps 3.0 → 3.5.** Existing v3 docs auto-upgrade on next backfill.

<specifics>
## Specific Ideas

- **User's real corpus is the proof point** — property docs where "Alex Doe owns Unit 204" is the exemplar triple.
- **Fixed predicate vocabulary prevents predicate explosion** — LLM can't invent "resides_at" vs "lives_in" vs "based_at" as 3 different relations.
- **Auto-inverse writes** — `A owns B` also stores `B owned_by A`. Users can query either direction.
- **Doc provenance per triple** — critical for user trust ("why does cortex think Alex owns Unit 204? → because of Sale Deed 2024.pdf").
- **User override + manual triples** — if LLM misses a relation, user adds it manually. Doesn't get overwritten by re-runs.
- **Ownership page groups by asset type** — Property / Vehicle / Investment / Business / Financial accounts. Matches how humans think about "my assets".

</specifics>
