# Phase 11.5 Plan 03: Pass3RelationExtractor Summary

**Async Rust relation extractor that mirrors Pass2LlmRefiner's semaphore/retry/fence-strip architecture to call the active LLM provider with doc text + Pass 2 entities and return Vec<Triple> gated by a 21-predicate closed vocabulary lock.**

## Performance

- **Duration:** ~20 min
- **Completed:** 2026-07-09
- **Tasks:** 2/2 completed
- **Files modified:** 2

## Accomplishments

- `src-tauri/src/pipeline/pass3_relation_extractor.rs` created with `Pass3Triple`/`Pass3Output` types, `parse_pass3_json` (two-attempt parse reusing Pass 2's `strip_json_fences`), and `validate_and_normalize` (drops unknown predicates, self-referential triples, out-of-bounds `e_N` references)
- `EXTRACT_RELATIONS_PROMPT` const declares all 21 predicates from `PREDICATE_VOCABULARY`, an explicit "NEVER invent new predicates" lock, and 4 few-shot examples (property sale deed, vehicle registration, identity doc, empty case) mirroring Pass 2's multi-region example style
- `Pass3RelationExtractor` struct with `new`/`new_with_capacity`/`model`/`set_model`/`semaphore`/`extract` lifecycle ŌĆö semaphore defaults to 8 in-flight (D-08), provider-absent and no-Pass2-entities short-circuits both return `Ok(None)` without any HTTP call (D-04)
- `extract()` maps validated `e_N` references to `CanonicalEntity` ids via `ExtractedEntity.canonical_id`, dropping any triple where either endpoint has no canonical id yet, and returns `Triple` records with empty `id`/`created_at`/`doc_ids` ready for `TripleStore.insert_one` to fill
- `pipeline/mod.rs` exports `pub mod pass3_relation_extractor;`
- 17 unit tests: JSON parse (direct + fence-stripped + invalid + empty-default), validation (unknown predicate / self-referential / out-of-bounds / valid-kept), prompt lock (all 21 predicates present + no-invention phrase), extractor lifecycle (no-provider, no-entities, semaphore default cap, semaphore concurrency timing, model get/set), and input chunking (short text with entity list, long text head/tail markers)

## Decisions Made

- Followed the plan's literal interface spec for `prepare_llm_input(title, text, pass2_entities, subject_map)` even though `subject_map: &[(String, String)]` is currently unused (prefixed `_subject_map`) ŌĆö keeps the signature stable for a future canonical-name-aware prompt enhancement without another breaking change
