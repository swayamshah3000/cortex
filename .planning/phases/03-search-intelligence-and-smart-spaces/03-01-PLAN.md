  Declare `pub mod query; pub mod highlight; pub mod filters;` and re-export key types.

  Actually, Rust module structure: create `src-tauri/src/search/mod.rs` as the module root, plus the three submodule files.

  **search/filters.rs — Pre-search metadata filtering (SRCH-03, SRCH-04):**
  - `apply_metadata_filters(filters: &SearchFilters, engine: &CortexEngine) -> Result<Option<HashSet<String>>, AppError>`
    - If all filter fields are None, return None (no filtering — search all).
    - If doc_type is Some: query filter_index.get("doc_type") for matching doc IDs.
    - If date_from/date_to is Some: query filter_index.get("created_at") for range.
    - If space_id is Some: query filter_index.get("space_ids") for matching.
    - If tags is Some: query filter_index.get("tags") for matching.
    - Intersect all non-None result sets.
    - Return the intersection as Some(HashSet<String>).
  - `parse_entity_filter(query: &str) -> Option<EntityFilter>` (SRCH-04)
    - Use regex to detect patterns like "invoices over $500", "documents before 2024-01-01"
    - Return EntityFilter { min_amount: Option<f64>, max_amount: Option<f64>, before_date: Option<String>, after_date: Option<String> }
    - This is best-effort NLP — extract obvious patterns, don't try to be perfect.
  - `apply_entity_filter(entity_filter: &EntityFilter, metadata: &HashMap<String, serde_json::Value>) -> bool`
    - Check extracted_entities in metadata against the parsed thresholds.

  NOTE: ruvector-filter's PayloadIndexManager may not support range queries directly. If so, implement filtering by iterating the collection's keys and checking metadata manually. The filter_index was set up with Keyword and Integer types in engine.rs. Check the actual API surface of PayloadIndexManager — if it only supports exact keyword match, use manual iteration with metadata comparison for date ranges and entity filters. The key design principle: filter BEFORE vector search to reduce the candidate set.

  **search/highlight.rs — Excerpt highlighting (SRCH-02):**
  - `find_best_excerpt(doc_text: &str, query_text: &str, window_size: usize) -> String`
    - Tokenize both query and document text into lowercase words.
    - Slide a window of `window_size` words across the document.
    - Score each window by counting query word overlaps.
    - Return the highest-scoring window as the excerpt string.
    - If no good match (score 0), return the first `window_size` words.
    - Default window_size: 30 words.
  - This is a simple word-overlap approach — good enough for v1, can be upgraded to TF-IDF later.

  **search/query.rs — Core search function (SRCH-01):**
  - `search_documents_impl(query: &str, filters: &SearchFilters, engine: &CortexEngine, embedding_service: &EmbeddingService) -> Result<Vec<SearchResult>, AppError>`
    1. Parse entity filters from query string (SRCH-04).
    2. Apply metadata filters to get candidate doc IDs (SRCH-03).
    3. Embed the query text using embedding_service.embed_text().
    4. Get the documents_384 collection from engine.collections.
    5. Perform HNSW search: use `collection.db.search(&query_vec, k)` or equivalent ruvector-core API.
       - k = 20 (return top 20 results).
       - If ruvector-core's search returns (id, distance) pairs, convert distance to score (1.0 - distance for cosine, or use similarity directly).
    6. If metadata filters returned a candidate set, intersect with search results (keep only IDs in the filter set).
    7. If entity filter was parsed, apply entity_filter to each result's metadata.
    8. For each surviving result, build a SearchResult:
       - Retrieve the full metadata from the vector entry (path, doc_type, size, entities, etc.).
       - Build a Document struct from the metadata.
       - Compute matched_excerpt using find_best_excerpt with the original document text from metadata["excerpt"] field (or re-read from disk if needed — prefer stored excerpt for performance).
       - Set score from the similarity value.
    9. Return results sorted by score descending.

  **Important API discovery:** Before coding, read `ruvector-core/src/vector_db.rs` and the collection type returned by `CollectionManager::get_collection()` to find the actual search/query method signature. The collection uses `collection.db` which is a VectorDB — look for a `search`, `query`, or `nearest` method. Also check `ruvector-core/src/index.rs` for the HNSW search API.

  Write unit tests:
  - Test `find_best_excerpt` with known text and query.
  - Test `parse_entity_filter` with "invoices over $500" and "documents before 2024-01-01".
  - Test `apply_metadata_filters` returns None when all filters are None.
