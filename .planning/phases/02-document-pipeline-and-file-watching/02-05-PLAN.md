  **Update watcher/worker.rs** -- Wire the file event handler to the real indexer:

  In the file events branch of the main loop, replace the placeholder with real indexing:
  - For CreateOrModify events:
    1. Emit `IndexProgress { status: "indexing", ... }`
    2. Clone the necessary Arcs (engine, embedding_service)
    3. Use `tokio::task::spawn_blocking` to call `indexer.index_file(&path, &engine, &embedding_service)` inside a blocking context
    4. The engine lock must be acquired INSIDE spawn_blocking via `engine.blocking_lock()`
    5. On success: emit `IndexProgress { status: "indexed", doc_id: Some(id), ... }`
    6. On skip (content hash matches): emit `IndexProgress { status: "skipped", ... }`
    7. On error: emit `IndexProgress { status: "error", error: Some(e.to_string()), ... }`

  NOTE: The indexer, embedding_service, and engine are all wrapped in Arc and can be cloned into the spawn_blocking closure. The indexer's index_file takes references -- clone the Arcs, move into the closure, then take references inside.

  CRITICAL: The EmbeddingService uses std::sync::Mutex internally, and index_file is synchronous. This means the entire call chain inside spawn_blocking is synchronous -- no async boundaries to worry about.

  For FWAT-06 (re-index on modification): This is already handled by the DocumentIndexer's content hash comparison logic from Plan 03. The watcher treats Create and Modify events the same way (CreateOrModify) -- both trigger index_file(), which internally checks the hash and skips if unchanged.

  #[tauri::command]
  pub async fn index_document(
      path: String,
      state: State<'_, AppState>,
  ) -> Result<DocumentMeta, AppError> {
      let engine = state.engine.clone();
      let embedding_service = state.embedding_service.clone();
      let indexer = state.indexer.clone();
      let path_owned = path.clone();

      let doc_id = tokio::task::spawn_blocking(move || {
          let path = std::path::Path::new(&path_owned);
          let engine = engine.blocking_lock();
          indexer.index_file(path, &engine, &embedding_service)
      }).await??;

      // Build DocumentMeta from the path and the generated doc_id
      let file_path = std::path::Path::new(&path);
      let file_meta = std::fs::metadata(&path).ok();
      Ok(DocumentMeta {
          id: doc_id,

              .unwrap_or("unknown")
              .to_string(),
          path,

      })

  **Update commands/folders.rs** -- Replace all folder command stubs:

  **add_watched_folder:**
  1. Lock `state.registry`, call `registry.add_folder(path.clone())`
  2. Save registry: `registry.save(&state.registry_path)?`
  3. Send `WatcherCommand::AddFolder { path, folder_id }` via `state.watcher_tx.send().await`
  4. Return `WatchedFolder` constructed from the config

  **remove_watched_folder:**
  1. Lock `state.registry`, get the folder config to retrieve the path
  2. Call `registry.remove_folder(&id)`
  3. Save registry
  4. Send `WatcherCommand::RemoveFolder { folder_id, path }` via `state.watcher_tx.send().await`

  **get_watched_folders:**
  1. Lock `state.registry`
  2. Map all `registry.folders.values()` to `WatchedFolder` structs
  3. Return the vec

  **trigger_scan:**
  1. Lock `state.registry`, get the folder config for the folder_id
  2. Walk the folder directory recursively using `std::fs::read_dir` (or `walkdir` if available -- use simple recursion if not)
  3. For each file: check exclusions, check type enabled
  4. For each matching file: call `indexer.index_file()` via spawn_blocking
  5. Emit `IndexProgress` events for each file
  6. Return `ScanProgress` with total/processed counts
  7. NOTE: trigger_scan should run in a background task (spawn the walk + indexing, return immediately with a "scanning" status). The progress comes via events.
