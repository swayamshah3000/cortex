<feature>
  <name>DocumentIndexer: Full pipeline orchestration</name>
  <files>src-tauri/src/pipeline/indexer.rs, src-tauri/src/pipeline/mod.rs</files>
  <behavior>
    The DocumentIndexer orchestrates the complete document ingestion pipeline.

    Cases:
    - New file (not previously indexed) -> parse, embed, extract entities, insert vector with metadata, return doc_id
    - Existing file with same content hash -> skip (return existing doc_id, no re-embedding)
    - Existing file with different content hash -> delete old vector, parse, embed, extract entities, insert new vector, return new doc_id
    - File with empty text after parsing -> return AppError::Parse("Document produced no text")
    - Unsupported file type -> return AppError::Parse from parser

    Metadata stored per vector:
    - path: String (absolute file path)
    - doc_type: String (pdf, docx, txt, etc.)
    - content_hash: String (SHA-256 hex)
    - extracted_entities: JSON array of ExtractedEntity
    - size: Number (file size in bytes)
    - title: String (filename)
    - excerpt: String (first 200 chars of parsed text)
    - created_at: String (file creation time ISO 8601)
    - modified_at: String (file modification time ISO 8601)
  </behavior>
  <implementation>
  Create `src-tauri/src/pipeline/indexer.rs`:

  **DocumentIndexer struct:**
  ```rust
  pub struct DocumentIndexer {
      extractor: EntityExtractor,
      path_index: std::sync::Mutex<HashMap<String, String>>,  // path -> doc_id
  }
  ```

  The `path_index` is an in-memory cache that maps file paths to document IDs, avoiding O(n) linear scans of the vector DB. It is populated at startup and updated on each index/delete operation.

  **new() constructor:**
  - Initialize EntityExtractor
  - Initialize empty path_index HashMap

  **rebuild_path_index(&self, collection: &Collection):**
  - Iterate all vector IDs via `collection.db.list_ids()` or similar
  - For each ID, `collection.db.get(&id)` -> extract "path" from metadata
  - Insert into path_index
  - Called once at startup to populate the cache
  - NOTE: Research mentions `db.keys()` -- use whatever API is available on VectorDB. If `keys()` doesn't exist, try `list()` or iterate. The important thing is to build the path->id map.

  **index_file(&self, path: &Path, engine: &CortexEngine, embedding_service: &EmbeddingService) -> Result<String, AppError>:**
  1. Compute `content_hash(path)?`
  2. Get collection: `engine.collections.get_collection("documents_384")`
  3. Check path_index for existing doc_id at this path
  4. If found, get the existing vector entry and compare stored content_hash
     - If same hash -> return existing doc_id (skip)
     - If different hash -> delete old vector via `db.delete(&existing_id)`, remove from path_index
  5. Parse document: `parser::parse_document(path)?`
  6. Validate: if `parsed.text.trim().is_empty()` -> return error
  7. Generate embedding: `embedding_service.embed_text(&parsed.text)?`
  8. Extract entities: `self.extractor.extract(&parsed.text)`
  9. Build metadata HashMap with all fields (path, doc_type, content_hash, extracted_entities as JSON, size from fs::metadata, title, excerpt = first 200 chars, created_at, modified_at)
  10. Create VectorEntry with `id: Some(Uuid::new_v4().to_string())`, vector, metadata
  11. Insert into collection (use `collection.read()` or `collection.write()` -- try read first since VectorDB may use internal locking; if it fails, use write)
  12. Update path_index with new mapping
  13. Return doc_id

  CRITICAL ANTI-PATTERNS TO AVOID:
  - Do NOT hold the collection lock during embedding (CPU-intensive). Parse and embed BEFORE acquiring the lock.
  - Do NOT hold a read lock then try to acquire write lock in the same scope -- deadlock. Drop the read guard first.
  - The embedding_service reference should NOT cross into the collection lock scope.

  Add `pub mod indexer;` to pipeline/mod.rs.
  </implementation>
</feature>
