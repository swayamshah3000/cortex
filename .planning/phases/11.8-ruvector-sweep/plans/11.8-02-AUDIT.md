# 11.8-02 Audit: `ruvector-attention` — Search Re-rank Integration

**Audited:** 2026-07-10
**Scope:** `crates/ruvector-attention/{README.md, src/lib.rs, src/traits.rs, src/attention/scaled_dot_product.rs, src/sparse/{flash.rs,local_global.rs}, src/moe/router.rs, Cargo.toml}`
**Decision gate:** D-08/D-09/D-11 (Phase 11.8 context)

## Verdict

NO-GO

## Fit-for-purpose analysis

Checking each D-08 criterion against the actual public API surface:

- **(a) Crate exposes a re-rank API taking query embedding + candidate embeddings + attention scores?**
  **FAIL.** Every mechanism implementing the public `Attention` trait (`ScaledDotProductAttention`, `MultiHeadAttention`, `FlashAttention`, `LinearAttention`, `LocalGlobalAttention`, `HyperbolicAttention`, `MixedCurvatureAttention`, `TopologyGatedAttention`, `DiffusionAttention`, etc.) implements the same signature:
  ```rust
  fn compute(&self, query: &[f32], keys: &[&[f32]], values: &[&[f32]]) -> AttentionResult<Vec<f32>>;
  ```
  This returns **one `Vec<f32>` of shape `[value_dim]`** — a softmax-weighted blend of the `values` vectors — not a `Vec<(usize, f32)>` or `Vec<f32>` of per-candidate relevance scores. There is no `rerank()`, `score()`, or equivalent public helper anywhere in the crate. Verified against `traits.rs` (the `Attention`, `GraphAttention`, `GeometricAttention`, `SparseAttention`, `TrainableAttention` trait definitions) — none expose a per-candidate scoring method.

  The one place per-candidate `(index, score)` pairs do exist internally — `LocalGlobalAttention::compute_local_scores` / `compute_global_scores` in `src/sparse/local_global.rs` — is a **private** (non-`pub`) helper consumed only inside `compute()`'s softmax-then-weighted-sum pipeline. It is not reachable from outside the crate.

  The only public method anywhere in the crate returning `Vec<(usize, f32)>` is `Router::route(&self, x: &[f32]) -> Vec<(usize, f32)>` in `src/moe/router.rs` (Mixture-of-Experts). This is semantically unrelated: it routes a *single* input vector to top-K of N *experts* (learned/trained routing weights), not a query against a list of *candidate document vectors*. Its signature doesn't even accept a `keys: &[&[f32]]` candidate list — wrong shape entirely, not adaptable without reimplementing scoring logic from scratch.

- **(b) At least one mechanism is CPU-friendly (no GPU requirement)?**
  **PASS (moot).** All 46 mechanisms are pure-Rust, CPU-only (SIMD via the `simd` default feature; no CUDA/Metal dependency anywhere in `Cargo.toml`). This criterion is satisfied but irrelevant given (a) fails.

- **(c) Attention scores are actionable (interpretable as relevance, not compression/dimensionality-reduction artifacts)?**
  **FAIL / N-A.** Because no mechanism exposes per-candidate scores publicly, there is nothing to interpret. The only artifact any `compute()` call yields is a blended `[dim]`-length vector — using it for reranking would require computing cosine-similarity between that blended vector and each candidate a second time, at which point ruvector-attention has added zero value over pure cosine (and burned the mechanism's O(n²) or O(n log n) compute cost for nothing).

**Overall: 1 of 3 criteria pass, and the passing one is moot given the blocking failure. This is a hard NO-GO — matches the exact "crate name doesn't match cortex-relevant functionality" failure pattern documented for `ruvector-cluster` in Phase 9.**

## Missing capabilities

1. No public API returns per-candidate relevance scores against a query — the crate's entire public surface (`Attention` trait and all 46 implementors) is built for the transformer use case (`compute(query, keys, values) -> blended_output_vector`), not the information-retrieval "score and rank N candidates" use case.
2. No `rerank(query, candidates, top_k) -> Vec<(idx, score)>` helper exists at any level (SDK, presets, or pipeline). The `sdk::AttentionPipeline` / `AttentionBuilder` composition layer (`add_norm`, `add_attention`, `add_residual`, etc.) is also built around the `compute()` output-vector shape — same problem, one layer up.
3. The only public `Vec<(usize, f32)>`-returning method (`Router::route`) is MoE expert-routing, not candidate-vector scoring — different input shape (`x: &[f32]` only, no candidate list) and different semantics (learned routing weights, not query-relevance).
4. To force-fit ruvector-attention into cortex's rerank use case would require: (a) writing brand-new code inside `attention_reranker.rs` that calls `compute()` per-candidate with `keys=[candidate], values=[candidate]` in a loop (defeating any batching/optimization the mechanism offers), then (b) computing cosine-similarity between each `compute()` output and the query to derive a usable relevance number — i.e., reimplementing cosine similarity ourselves while paying the mechanism's overhead. This is strictly worse than not using the crate at all.

## Fallback recommendation

Cortex ships without ruvector-attention-based re-ranking. The existing blended score formula in `search_documents_impl` (`src-tauri/src/search/query.rs`) stands as-is:

```rust
let score = (0.9 * cosine + 0.1 * recency).clamp(0.0, 1.0);
```

Per D-12, no `0.5*cosine + 0.3*attention + 0.15*jaccard + 0.05*recency` blend is implemented. Plan 07 (attention re-ranker wiring) becomes a documented deviation/skip: `11.8-07-DEVIATION.md` should reference this audit and confirm the wave that would have consumed Plan 07 is dropped. No new `search/attention_reranker.rs` module is created, no Cargo path-dependency on `ruvector-attention` is added to `src-tauri/Cargo.toml`, and no Settings toggle for "Enhanced search re-ranking (experimental)" (D-13) ships in this phase.

If real query-relevance reranking is desired in the future (v1.3+), the correct building block is NOT ruvector-attention as currently published — it would require either (a) a future ruvector-attention release that adds a public per-candidate scoring API, or (b) implementing a lightweight in-house cross-encoder-style reranker (e.g. late-interaction dot-product scoring over the existing HNSW candidate embeddings) directly in cortex, which doesn't need this crate at all.

## Rationale for mechanism choice

N/A — no mechanism was selected. The audit blocks at criterion (a): the crate has no API shape compatible with "take a query + N candidates, return N relevance scores." All 46 mechanisms were surveyed at the trait level (`Attention`, `GraphAttention`, `GeometricAttention`, `SparseAttention`, `TrainableAttention` in `traits.rs`, plus the concrete top-3 CPU-friendly/semantic-relevance-oriented candidates inspected in detail — `ScaledDotProductAttention`, `FlashAttention`, `LocalGlobalAttention`) and all share the identical `compute() -> Vec<f32>` (blended value vector) return shape. Since the failure is structural (API shape, not implementation quality of any one mechanism), enumerating mechanism-specific latency/quality tradeoffs was not pursued further — no mechanism, however fast, can be substituted for a scoring API that does not exist.
