# 11.8-03 Audit: ruvector-solver for Entity PageRank

**Audited:** 2026-07-10
**Crate:** `/Users/gshah/work/apps/experiments/ruvector/crates/ruvector-solver/` (v0.1, MSRV 1.77)
**Purpose:** Determine GO/NO-GO for D-14/D-15/D-16 — replacing `doc_count DESC` prominence
signal in Sidebar/Ontology panel with graph-importance ranking (PageRank) over the
TripleStore.

## Verdict

**NO-GO**

## Fit-for-purpose analysis

- **(a) Crate exposes PageRank on user-supplied sparse graph?** Partially, but not the kind
  needed. `ruvector-solver` implements **Personalized PageRank (PPR)** via the Andersen-Chung-Lang
  Forward Push algorithm (`forward_push.rs`), Backward Push (`backward_push.rs`), and Hybrid
  Random Walk (`random_walk.rs`). All three require a **seed vertex or seed distribution** as
  input and return scores *relative to that seed* — not a single global importance score per
  node across the whole graph. The crate's own `QueryType` enum (`types.rs:440-471`) only
  offers `PageRankSingle { source }` and `PageRankPairwise { source, target }`. There is no
  `PageRankGlobal` / uniform-teleport whole-graph variant anywhere in the crate (confirmed via
  `rg -i "page.?rank"` across all of `src/` — every hit is PPR/single-source/pairwise). The
  crate's own README table explicitly lists "3 sublinear algorithms" for "PageRank" but all
  three are PPR variants (Forward Push, Backward Push, Hybrid Random Walk), not classic
  Brin-Page global PageRank.
- **(b) Accepts `Vec<(subject_id, object_id, weight)>` edge list (or planner can adapt via
  ID→index map)?** Yes, adaptable. `CsrMatrix::from_coo(rows, cols, entries: impl
  IntoIterator<Item = (usize, usize, f64)>)` (`types.rs:330`) accepts COO triplets directly.
  Plan 08 would need to build a `String -> usize` ID map over `TripleStore.forward_index` /
  `reverse_index` keys first (entity IDs are `String`, not `usize`), then map back after solve.
  This part of the integration is straightforward and would work.
- **(c) Returns `Vec<(node_id, rank_score)>` (or adaptable equivalent)?** Yes for PPR:
  `ForwardPushSolver::ppr_from_source(&self, graph, source) -> Result<Vec<(usize, f64)>,
  SolverError>` returns exactly this shape, sorted descending by score. The shape is right; the
  semantics are wrong for the use case (see below).
- **(d) Sublinear O(log n) is claimed (per D-14) — is this actually documented, or is it
  O(iterations × edges)?** The crate's complexity claims are real but apply to PPR only:
  Forward Push is `O(1/epsilon)` total pushes, independent of graph size (confirmed in both
  README and `forward_push.rs` doc comments + `estimate_complexity` returning
  `ComplexityClass::SublinearNnz`). This sublinearity is a *per-source-query* guarantee. To
  approximate a "global importance" ranking Plan 08 would need to run PPR once per candidate
  seed (or use every node as a seed and aggregate), which destroys the sublinear win — it
  becomes O(n / epsilon), i.e. linear in graph size times a constant, not O(log n) globally as
  D-14 assumed.

## Why NO-GO (not GO-WITH-CAVEAT)

D-16 requires: "Runs on TripleStore's forward-index. Edge weight = triple provenance doc
count. Result: `HashMap<entity_id, pagerank_score>`" — i.e., **one importance score per entity,
comparable across all entities, with no notion of "relative to a seed."** That is exactly
classic global PageRank (uniform random restart across all nodes), which this crate does not
implement. Every algorithm in `ruvector-solver` is seed-relative PPR. Adapting PPR into a
global ranking would require either:

1. Running `ppr_from_source` once per entity as source and combining results (defeats
   sublinearity, O(n) separate solves, and the resulting "average PPR rank" is not
   mathematically equivalent to global PageRank — it approximates something closer to
   eigenvector centrality of the *transpose*, with subtly different convergence properties), or
2. Using `ppr_multi_seed` with **all nodes as seeds with uniform weight `1/n`** — this is the
   closest analog to true global PageRank (uniform teleport = global PageRank's damping
   formulation) and is mathematically closer to correct, but is untested for this purpose,
   undocumented as an intended use case by the crate authors, and still costs `O(n/epsilon)`
   for the seed injection, not sublinear per D-14's assumption. It would also need careful
   validation against a reference PageRank implementation before trusting it as a UI-facing
   "who matters" signal — out of scope for an audit-only plan.

Given the Phase 9/12/13 lesson this audit gate exists to enforce — "ruvector crate names
sometimes don't match cortex-relevant functionality" — this is precisely that pattern: the
crate is *named* and *marketed* around PageRank, but the actual algorithm family (PPR) solves
a materially different problem than the one D-14 needs (global entity importance ranking).

## Missing capabilities

- No global/whole-graph PageRank (uniform teleport across all nodes with no seed bias).
- No `QueryType::PageRankGlobal` or equivalent in the router.
- `ppr_multi_seed` with uniform all-node seeding is a theoretically plausible workaround but
  is unvalidated, expensive (O(n) seed injection defeats the "sublinear" selling point), and
  would require its own correctness audit (compare against a reference PageRank on a known
  graph) before being trusted for a user-visible ranking — that validation work is out of
  scope for Plan 08 as currently scoped (implementation-only, no algorithm R&D).

## Fallback recommendation

Per D-15: skip this integration. Plan 08 (`graph/entity_pagerank.rs`) does not get built.

- **Sidebar top-5 prominent entities** continues to use `doc_count DESC` (existing behavior
  from Phase 11.6's `get_prominent_entities` IPC). No regression — this is already shipped and
  works.
- **Ontology panel prominence** remains unchanged (doc_count-based).
- Write `.planning/phases/11.8-ruvector-sweep/plans/11.8-08-DEVIATION.md` (or equivalent) when
  Plan 08 is reached, pointing back to this audit as the reason for skipping, so Plan 08's
  executor does not re-litigate this analysis.
- **Future option (v1.3+, out of scope now):** if global entity-importance ranking is desired
  later, evaluate implementing a small in-house power-iteration PageRank directly (classic
  Brin-Page formulation is ~30 lines: `r' = alpha * M @ r + (1-alpha)/n`, iterate to
  convergence) rather than forcing `ruvector-solver`'s PPR primitives into a role they were not
  designed for. At Cortex's current graph scale (single-user desktop, thousands of entities,
  not millions), a dense/simple sparse power-iteration in pure Rust would be fast enough
  without needing any external solver crate at all — likely simpler and more correct than
  adapting PPR.
