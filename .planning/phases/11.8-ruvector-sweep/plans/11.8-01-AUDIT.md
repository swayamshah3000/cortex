# 11.8-01: ruvllm Crate Audit

**Audited:** 2026-07-10
**Auditor:** GSD executor (Plan 11.8-01)
**Scope:** `ruvector/crates/ruvllm/{README.md, src/lib.rs, src/backends/{mod.rs, candle_backend.rs}, src/tokenizer.rs, Cargo.toml}`, `tests/real_model_test.rs`

## Verdict

GO

`ruvllm` exposes a directly usable `LlmBackend` trait (`generate`, `generate_stream_v2`, `load_model`) via `CandleBackend`, with Metal acceleration on Apple Silicon, GGUF loading, chat-template formatting, and both sync and async streaming. The API surface maps cleanly onto cortex's existing `ai_request(AIServiceRequest) -> AIServiceResponse` abstraction (Phase 7) with no architectural rework required.

## Fit-for-purpose analysis

| D-01 Criterion | Status | Evidence |
|---|---|---|
| (a) Rust crate loadable via Cargo path dep | PASS | `ruvllm/Cargo.toml` is a standard workspace member (`crate-type = ["rlib"]`). Cortex can add `ruvllm = { path = "../../../experiments/ruvector/crates/ruvllm", default-features = false, features = [...] }` exactly like the existing `ruvector-*` deps already used elsewhere in cortex's Cargo.toml. |
| (b) Anthropic-style chat completion API surface exists or is adaptable | PASS | `LlmBackend::generate(&self, prompt: &str, params: GenerateParams) -> Result<String>` is a direct sync call. Multi-turn chat is handled via `tokenizer::{ChatMessage, Role}` + `CandleBackend::apply_chat_template(&[ChatMessage]) -> Result<String>`, which flattens `system_prompt` + `Vec<ServiceMessage>` into a single formatted prompt string before calling `generate()`. This is a straightforward adapter, not a rewrite. |
| (c) Metal backend available on Apple Silicon | PASS | `CandleBackend::with_device(DeviceType::Metal)` is first-class; `DeviceType::Metal` is the `#[default]` variant of the enum (`backends/mod.rs:372`). Feature flags `metal`, `inference-metal`, `inference-metal-native` wire `candle-core/metal` + native Metal compute shaders. README documents M4 Pro benchmarks (Qwen2.5-7B Q4K: 95 tok/s decode, 4.2GB memory). |
| (d) Streaming supported | PASS | Two streaming APIs exist: (1) sync `generate_stream_v2(&self, prompt, params) -> Result<TokenStream>` where `TokenStream` implements `Iterator<Item = Result<StreamEvent>>` with `StreamEvent::{Token(GeneratedToken), Done{..}, Error(String)}`; (2) async `LlmBackendAsync::generate_stream_async(&self, prompt, params) -> Result<AsyncTokenStream>` (blanket-impl'd for any `LlmBackend`, gated behind the `async-runtime` feature which is in `default`). `AsyncTokenStream` implements `futures_core::Stream`, directly composable with cortex's existing `ai_request_stream()` (Phase 11.7). |

**Additional fit notes:**
- `create_backend() -> Box<dyn LlmBackend>` gives a feature-gated factory (falls back to `NoopBackend` if `candle` disabled) — useful for graceful init-failure handling (D-06 fallback requirement).
- `ModelConfig::device: DeviceType` + `ModelArchitecture::detect_from_model_id()` auto-detects Llama/Mistral/Qwen/Phi/Gemma family from the HF model-id string — reduces manual config for the "select model" UX (D-04).
- `LlmBackend::load_model(&mut self, model_id: &str, config: ModelConfig)` transparently handles three input forms: (1) direct `.gguf` file path, (2) local directory with GGUF/safetensors + `tokenizer.json`, (3) HuggingFace Hub model ID string (auto-downloads via `load_from_hub`). This directly satisfies D-04's "model download UX" requirement — no separate downloader integration needed; passing an HF repo id to `load_model()` triggers download automatically.
- `hub::{ModelDownloader, DownloadConfig, DownloadProgress, ProgressCallback}` gives an explicit progress-bar hook if cortex wants a dedicated download-progress UI (D-04) rather than relying on the implicit hub download inside `load_model`.
- Errors are a clean `RuvLLMError` enum (`Model`, `OutOfMemory`, `Generation`, `Ane`, `Gguf`, `Config`, `Storage`, `NotFound`, `InvalidOperation`) — maps to cortex's `Result<AIServiceResponse, String>` via `.to_string()`, consistent with how `anthropic_chat`/`openai_chat` already return `String` errors.

**Cargo dependency conflict check:**
- `tokio`: cortex uses `tokio = { version = "1", features = ["full"] }`; ruvector workspace uses `tokio = { version = "1.41", features = ["rt-multi-thread", "sync", "macros"] }`. Both resolve to the same `1.x` semver-compatible line — Cargo will unify to a single `tokio` build. No conflict.
- `reqwest`: `ruvllm` itself does not depend on `reqwest` directly (network access for hub downloads goes through `hf-hub`, which has its own internal HTTP client, feature-gated with `["tokio"]`). No collision with cortex's `reqwest` usage in `ai/service.rs`.
- `serde`/`serde_json`: both use workspace-inherited versions; no conflict expected (both are `1.x` line crates in wide use).
- `candle-core`/`candle-nn`/`candle-transformers` (`0.8`), `tokenizers` (`0.20`), `hf-hub` (`0.3`) are new transitive deps ruvllm brings in — cortex does not currently depend on these, so no version collision, but they will materially increase the compiled binary size and cold build time (Candle + its Metal kernels are a non-trivial dependency tree).
- macOS-only deps (`metal`, `objc`, `objc2*`, `block2`) are already gated behind `target_os = "macos"` + feature flags — safe to enable unconditionally on this project since the target platform is Apple Silicon macOS.

## Target API surface

Adapter sketch — new `src-tauri/src/ai/ruvllm.rs` function callable from `ai_request()`, mirroring the existing `ollama_chat` / `anthropic_chat` signature pattern already used in `ai/service.rs`:

```rust
use ruvllm::{
    backends::{CandleBackend, DeviceType, GenerateParams, LlmBackend, ModelConfig},
    tokenizer::{ChatMessage, Role},
};

pub async fn ruvllm_chat(
    backend: &CandleBackend,           // held in Tauri AppState, model already loaded
    system_prompt: &str,
    messages: &[ServiceMessage],
    max_tokens: u32,
) -> Result<AIServiceResponse, String> {
    let mut chat = vec![ChatMessage::system(system_prompt)];
    chat.extend(messages.iter().map(|m| match m.role.as_str() {
        "assistant" => ChatMessage::assistant(m.content.clone()),
        _ => ChatMessage::user(m.content.clone()),
    }));

    let prompt = backend
        .apply_chat_template(&chat)
        .map_err(|e| format!("ruvllm chat template error: {e}"))?;

    let params = GenerateParams::default()
        .with_max_tokens(max_tokens as usize)
        .with_temperature(0.7);

    let text = backend
        .generate(&prompt, params)
        .map_err(|e| format!("ruvllm generation error: {e}"))?;

    Ok(AIServiceResponse {
        content: text,
        model: backend.model_id().to_string(),
        input_tokens: None,   // ruvllm ModelInfo does not expose per-call token counts today
        output_tokens: None,
    })
}
```

Model init (one-time, at "Connect" or app startup when `local-ruvllm` is active provider):

```rust
let mut backend = CandleBackend::with_device(DeviceType::Metal)
    .map_err(|e| format!("Metal init failed: {e}"))?;
backend.load_model(
    "Qwen/Qwen2.5-7B-Instruct-GGUF",   // or a local .gguf path in app_data_dir/ruvllm-models/
    ModelConfig { quantization: Some(Quantization::Q4K), device: DeviceType::Metal, ..Default::default() },
)?;
```

Streaming variant for RAG chat (Phase 11.7's `ai_request_stream()`), using the async wrapper:

```rust
use ruvllm::backends::{LlmBackendAsync, StreamEvent};

let stream = backend.generate_stream_async(&prompt, params).await
    .map_err(|e| format!("ruvllm stream init error: {e}"))?;
// stream implements futures_core::Stream<Item = Result<StreamEvent>>
// forward StreamEvent::Token(t).text chunks to the existing SSE/Tauri-event sink used for cloud providers
```

**Struct/method chain summary:** `CandleBackend::with_device(DeviceType::Metal)? -> backend.load_model(model_id, ModelConfig{..})? -> backend.apply_chat_template(&messages)? -> backend.generate(prompt, params)?` (sync) or `backend.generate_stream_async(prompt, params).await?` (async streaming).

## Recommended Cargo features

For Apple Silicon (M-series), add to `src-tauri/Cargo.toml`:

```toml
ruvllm = { path = "../../experiments/ruvector/crates/ruvllm", default-features = false, features = ["inference-metal", "gguf-mmap", "parallel", "accelerate"] }
```

- `inference-metal` — pulls in `candle` + `metal`, the minimum needed for GPU-accelerated Candle inference on Apple Silicon.
- `gguf-mmap` — memory-mapped GGUF loading (fast load, lower peak RAM during load).
- `parallel` — Rayon-based multi-threaded GEMM/GEMV (4-6x speedup per README, relevant on the user's M-series multi-core CPU for any CPU-side ops).
- `accelerate` — Apple Accelerate BLAS, ~2x GEMV speedup, macOS-only, free to enable.
- Explicitly **not** recommending `coreml`/`hybrid-ane`/`metal-compute` at v1 — those add ANE routing complexity and native Metal shader compilation; the plain `inference-metal` Candle path is the safer, better-tested default per the "Backends" comparison table (Candle = "Single user, edge, WASM" is the exact cortex use case). Can be revisited in v1.3 if `inference-metal` benchmarks are copy short.
- `async-runtime` is already in `default` (not disabled since `default-features = false` — must re-add explicitly if kept minimal): add `"async-runtime"` to the feature list above if streaming (`generate_stream_async`) is needed, which it is per D-05. Corrected feature list:

```toml
ruvllm = { path = "../../experiments/ruvector/crates/ruvllm", default-features = false, features = ["inference-metal", "gguf-mmap", "parallel", "accelerate", "async-runtime"] }
```

## Recommended default model

**Qwen2.5-7B-Instruct, Q4_K_M GGUF quantization.**

- Rationale: best decode throughput/quality tradeoff in the README's own M4 Pro benchmark table (95 tok/s decode, 4.2 GB memory footprint) among the 7B-class options; Qwen 2.5 has strong instruction-following and multilingual support suited to the varied personal-document corpus cortex indexes (entity extraction, space naming, RAG chat).
- HuggingFace repo: `Qwen/Qwen2.5-7B-Instruct-GGUF` (community GGUF quants widely available; exact quant file `qwen2.5-7b-instruct-q4_k_m.gguf`, ~4.4-4.7GB download).
- Secondary/lightweight option for lower-spec Macs or first-run default: **Llama 3.2 3B Instruct** GGUF Q4_K_M (~2GB download, faster cold start, acceptable quality for entity extraction / space naming tasks that don't need deep reasoning). Recommend exposing both in the D-04 model-selection dialog, matching the context doc's own model-picker language ("Qwen 2.5 7B / Llama 3.2 3B").
- RuvLTRA-Small (494M, purpose-built for this ecosystem) is an interesting fallback for extremely constrained hardware but is not proven for general-purpose entity extraction / RAG quality — do not default to it; leave as a future opt-in.

## Streaming

YES. Streaming variant: `LlmBackend::generate_stream_v2()` (sync, returns `TokenStream: Iterator<Item = Result<StreamEvent>>`) and its async counterpart `LlmBackendAsync::generate_stream_async()` (returns `AsyncTokenStream: futures_core::Stream<Item = Result<StreamEvent>>`, blanket-implemented for every `LlmBackend`, feature-gated on `async-runtime` which must be included in the Cargo feature list above). `StreamEvent::Token(GeneratedToken)` carries per-token text; `StreamEvent::Done { total_tokens, duration_ms, tokens_per_second }` signals completion with built-in throughput metrics — directly usable for the D-07 Settings "last query 1.2s" benchmark display.
