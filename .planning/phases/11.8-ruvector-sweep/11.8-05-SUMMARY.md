# Phase 11.8 Plan 05: ruvector-hyperbolic-hnsw Wiring Summary

**Wires the Phase 10 hyperbolic HNSW secondary index into `search_documents_impl` — parent-scoped queries now route through hierarchy-aware hyperbolic validation when the index is populated (post-recluster), and fall back silently to the pre-existing flat HNSW + Space-membership filter otherwise; non-scoped queries are provably byte-identical to Phase 10.**

## Accomplishments

- Added `space_descendant_candidates(space_manager, space_id) -> HashSet<String>` in `search/filters.rs`: recursively unions a parent Space's own `doc_ids` with all its `sub_space_ids`' doc_ids. Returns empty set (no panic) for unknown space_id.
- Added `SpaceManager::insert_space_data_for_test` (test-only, `#[cfg(test)]`) to seed `SpaceData` directly in unit tests without running the full async `recluster()` pipeline.
- Added `hyp_search(query_vec, k, hyp_index: Option<&HyperbolicHnsw>, hyp_id_to_space: &[String]) -> Vec<String>` in `search/query.rs`: queries the secondary hyperbolic index and maps internal `usize` ids back to `space_id` strings. Returns `Vec::new()` on `None` index, empty id map, or search error (D-21 silent fallback) — bounds-checked, never panics on out-of-range ids.
- Extended `search_documents_impl` with 3 new parameters (`hyp_index`, `hyp_id_to_space`, `space_manager`) and a new Step 2c: when `filters.space_id.is_some()`, intersect `candidate_set` with `space_descendant_candidates`, and — when a populated hyperbolic index is available — additionally resolve the requested space's top-level ancestor and confirm it's a top-K hyperbolic hit (routing validation, not narrowing).
- Wired the IPC caller `commands/documents.rs::search_documents`: acquires short-lived `blocking_lock()` guards on `state.hyp_index`, `state.hyp_id_to_space`, `state.space_manager` inside the existing `spawn_blocking` closure — never held across an `.await`, mirroring the pre-existing `entity_store_guard` pattern.
- 7 new unit/integration tests: 3 for `space_descendant_candidates` (recursive union, unknown-space-empty, leaf-sub-space), 4 for `hyp_search` (None-index, empty-id-map, k=3-of-5-ordered, k=1-single-space-self-hit) — all run in the default `cargo test --lib` (fast, no external deps).
- 4 additional integration tests directly on `search_documents_impl` (non-scoped byte-identical regression guard, scoped+hyp-None flat fallback, scoped+hyp-populated hierarchy-validated routing, top-level self-hit via `hyp_search`) — marked `#[ignore]` per the existing codebase convention (`pipeline/indexer.rs`) since they require a real `EmbeddingService::new_local()` (fastembed model). **Verified passing** with the locally-cached model (`.fastembed_cache/`) during this execution — see Verification section.
- Re-ran the Phase 10 SC5 perf gate (`test_sc5_hierarchical_search_perf_gate`, `--release --ignored`): **PASSED** — flat baseline 5480ms (10K-doc `cluster_documents`, k=20) vs hyperbolic search 0ms (single ANN query, k=10). Ratio is effectively 0×, far under the ≤2× ceiling. No regression from Phase 10.

## Decisions Made

See `key-decisions` in frontmatter. Summary:
1. Extra-params signature extension (not a bundled `HypCtx` struct) — consistent with existing `entity_store: &EntityStore` param style.
2. Sub-space scoping: exact-descendant narrowing always via `space_descendant_candidates` on the ACTUAL requested space; hyperbolic validation resolves to the top-level ancestor (hyp_index only holds depth==0 centroids) and is confirmation-only, never re-narrows candidates.
3. `hyp_search` takes `Option<&HyperbolicHnsw>` (borrowed) rather than `&Option<HyperbolicHnsw>` — avoids a `.clone()` at the call site even though the type is `Clone`-derived; no functional difference, cleaner ownership.
4. No `SearchFilters` struct change — `space_id` (not `parent_space_id`) is the existing field already wired to this purpose since Phase 9/10.

Used the real struct field names (`r.id`) rather than tuple destructuring. No behavior difference, just accurate to the actual crate surface.
