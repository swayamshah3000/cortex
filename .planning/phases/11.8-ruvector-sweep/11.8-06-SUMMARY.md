# Phase 11.8 Plan 06: ruvllm Local LLM Provider Summary

**Integrated ruvllm as the `local-ruvllm` provider — Metal-accelerated Qwen2.5-7B-Instruct inference via CandleBackend, routed through the existing ai_request()/ai_request_stream() abstraction with zero consumer-code changes, plus a download_ruvllm_model Tauri command for first-run model acquisition.**

## AUDIT Verdict + How It Was Honored

**Verdict: GO** (11.8-01-AUDIT.md, verified 2026-07-10).

The audit's confirmed target API surface —
`CandleBackend::with_device(DeviceType::Metal)? -> backend.load_model(model_id, ModelConfig{..})? -> backend.apply_chat_template(&messages)? -> backend.generate(prompt, params)?` —
was implemented verbatim in `ai/ruvllm.rs::ruvllm_chat`, and cross-checked directly against the real ruvllm crate source (`backends/mod.rs`, `backends/candle_backend.rs`, `tokenizer.rs`) before writing code, confirming every struct/method name, signature, and feature flag in the audit matches what actually compiles. `cargo check`, `cargo test --lib -- ruvllm` (9 tests), `cargo build`, and `cargo build --release` (4m54s, LTO+opt-level=s) all succeed.

## Exact ruvllm API Calls Used

- `CandleBackend::with_device(DeviceType::Metal)` — backend init
- `LlmBackend::load_model(&mut self, model_id: &str, config: ModelConfig)` — model load (also triggers `load_from_hub()` internally for HF repo ids)
- `CandleBackend::apply_chat_template(&self, messages: &[ChatMessage]) -> Result<String>` — system+multi-turn prompt flattening
- `LlmBackend::generate(&self, prompt: &str, params: GenerateParams) -> Result<String>` — sync completion
- `LlmBackend::generate_stream_v2(&self, prompt: &str, params: GenerateParams) -> Result<TokenStream>` — sync streaming (`Iterator<Item = Result<StreamEvent>>`)
- `CandleBackend::model_id(&self) -> &str` — response `model` field
- `ChatMessage::system/user/assistant` + `Role` — message construction
- `ModelConfig { quantization: Some(Quantization::Q4K), device: DeviceType::Metal, ..Default::default() }`
- `GenerateParams::default().with_max_tokens(n).with_temperature(0.7)`

## Default Model Shipped

**Qwen/Qwen2.5-7B-Instruct-GGUF**, Q4_K_M quantization — matches 11.8-01-AUDIT.md "Recommended default model" exactly (95 tok/s decode / 4.2GB memory on M4 Pro per the crate's own README benchmarks). Fallback/lightweight option: `meta-llama/Llama-3.2-3B-Instruct-GGUF`. Both are in the `allowed_model_ids()` allowlist enforced by `download_model()`.

## Progress Event Name

`ruvllm://download-progress` — payload `{modelId: string, bytesDownloaded: number, totalBytes: number, status: "downloading"|"complete"|"error", error: string|null}` (camelCase on wire via `#[serde(rename_all = "camelCase")]`). Emitted at start (`0/1`), on completion (`1/1`), and on error.

## Fallback Error Prefix

`[FALLBACK]` — applied to: Metal device-init failure, model-load failure, backend-not-initialized state, spawn_blocking join errors, and the defensive `service.rs` match arm. Plan 10's UI should treat a `[FALLBACK]`-prefixed error as "swap to previously active provider + toast."

**Additional signal for Plan 10:** the single most common runtime-init state — model not yet downloaded on first run — returns a plain-text error containing the substring `"not downloaded"` (and names the Settings path explicitly) rather than the `[FALLBACK]` prefix, since this is an *expected* first-run state (the user simply hasn't clicked "Download model" yet), not a genuine failure requiring provider fallback. Plan 10 should detect **both** `"[FALLBACK]"` prefix and `"not downloaded"` substring and route them to slightly different UX (fallback+toast vs. "please download the model" prompt).

## Streaming Caveats

Streaming IS implemented and functional (`ruvllm_chat_stream`), using ruvllm's sync `generate_stream_v2()` iterator (not the async `generate_stream_async()` wrapper), because the `CandleBackend` handle is held behind a plain `tokio::sync::Mutex<Option<CandleBackend>>` for simplicity — the sync iterator is drained inline while the async fn holds the lock, forwarding each `StreamEvent::Token` to a caller-supplied `FnMut(&str)` callback. This is fully functional for Phase 11.7's RAG chat integration, not a stub. A future iteration could switch to `generate_stream_async()` + `futures_core::Stream` composition for true async yielding if profiling shows the lock-held-during-generation pattern causes contention with concurrent requests (out of scope here — Pass 2/3 backfill and RAG chat are not expected to run concurrently against local-ruvllm per D-23's provider-aware concurrency semaphore, which stays owned by a downstream plan).

## Decisions Made

- Held the `CandleBackend` behind a module-local lazy singleton (`std::sync::OnceLock<Arc<Mutex<Option<CandleBackend>>>>`) rather than adding it to `AppState`, because `ai_request()`'s signature (`&AuthState` only) does not have access to `AppState`, and changing that signature would ripple through every existing call site (`commands/ai.rs::chat`, `commands/ai.rs::test_connection`, `ai/retry.rs`) — out of scope for this plan per its "no consumer code changes" objective.
- Derived `app_data_dir` from `AuthState.store_path.parent()` (the directory containing `credentials.json`) rather than adding a new parameter to `ai_request`, for the same reason.
- A `.ready` marker file at `model_storage_path(app_data_dir, model_id)/.ready` is written after a successful `load_model()` call, so `ruvllm_chat` can give an immediate, clear "not downloaded" error instead of silently kicking off a multi-GB implicit download on every chat call before a model has been explicitly downloaded via Settings.

### Documented (not auto-fixed — see plan's own deviation_rules)

**3. Model-download progress is coarse-grained, not byte-level streaming**
- ruvllm's `load_model()` downloads GGUF/tokenizer files via `hf_hub::api::sync::Api` (a synchronous HTTP client) internally when given an HF repo id — this path does not expose a byte-level progress callback that Task 3's literal spec ("emits progress events at least once per second... bytesDownloaded/totalBytes") requires. Implementing true byte-level progress would require either (a) vendoring a custom async downloader on top of raw `reqwest` that duplicates `load_from_hub`'s file-discovery logic (GGUF filename pattern matching per quantization), or (b) modifying the `ruvllm` crate itself to add a progress hook to `load_from_hub` — both are out of scope for a single-plan execution pass. `download_model()` instead emits two coarse steps: `(0, 1)` at start, `(1, 1)` on completion, with `status: "downloading"` in between via the Tauri command wrapper. Plan 10's UI should render this as an indeterminate/spinner progress state rather than a determinate byte-count bar for `local-ruvllm` specifically.
- **GGUF filename mismatch risk:** `load_from_hub`'s internal filename search tries `model-q4_k_m.gguf`, `model.Q4_K_M.gguf`, `ggml-model-q4_k_m.gguf` for `Quantization::Q4K` — these may not match the actual filename convention used by `Qwen/Qwen2.5-7B-Instruct-GGUF`'s HF repo (commonly `qwen2.5-7b-instruct-q4_k_m.gguf` per the audit's own notes). If the exact filename doesn't match, `load_from_hub` falls back to loading safetensors instead of GGUF, which is a heavier download and different code path. This was not verified against a live network call in this execution pass (no network access / no desire to trigger a multi-GB download during automated testing) — flagged for manual verification during the first real "Download model" click in Settings (Plan 10).
